Three-Tier Architecture

The default Spring Initializer structure is not wrong — it just needs discipline to remain sustainable. These are the rules that make it extensible without requiring an architectural overhaul.

Presentation Layer Controllers · Request/Response models · @Valid
↓   dependencies point downward   ↓
Service Layer Business logic · Orchestration · DTOs
↓
Repository Layer Persistence · JPA Entities · Spring Data
Core rule: Dependencies point downward only. Controllers know about services. Services know about repositories. Nothing points back up.
// Section 02

Layer Responsibilities

Each layer has exactly one job. When logic ends up in the wrong layer, the seams break and the architecture becomes cosmetic.

01
Controller handles HTTP only
Deserialisaton, validation, routing, serialisation. No business rules. No repository calls.
02
Service contains business logic
Orchestrates, applies rules, coordinates repositories. Knows nothing of HTTP or JSON.
03
Repository handles persistence only
Queries, saves, and deletes. No business rules. Returns entities — not DTOs.
04
Entity protects its own state
Business rules live in the model. Public setters for state transitions are a warning sign.
05
No 1:1 controller-to-service rule
A controller can call multiple services. Multiple controllers can share one service. Follow the domain, not the scaffold.
06
Map at every boundary
Translate between types as you cross layers. Never pass an entity to the controller or a request model to the repository.
Why not call repositories directly from controllers?

Bypassing the service layer collapses your business logic into the controller. When a second controller needs the same logic you duplicate it. When the logic changes you have to find every caller. The service layer is the single point of truth for business rules — removing it removes the seam.

// ❌ Don't
@GetMapping("/{id}")
public OrderResponse getOrder(@PathVariable Long id) {
    Order entity = orderRepository.findById(id).orElseThrow();
    return new OrderResponse(entity.getId(), entity.getStatus());
}

// ✅ Do
@GetMapping("/{id}")
public OrderResponse getOrder(@PathVariable Long id) {
    return orderService.getOrder(id);
}
How many services can one controller call?

As many as the use case requires. A checkout endpoint might reasonably orchestrate an order service, payment service, and notification service. The controller is an entry point, not a 1:1 proxy.

@PostMapping("/orders/{id}/checkout")
public CheckoutResponse checkout(@PathVariable Long id) {
    Order order     = orderService.getOrder(id);
    PaymentResult r = paymentService.charge(order);
    notificationService.confirmOrder(order, r);
    return new CheckoutResponse(order, r);
}

Distinct Types at Every Boundary

Four distinct type categories, each owned by a specific layer. Conflating them creates invisible coupling between layers that should be independent.

Request Model
Presentation layer inbound
record PlaceOrderRequest(
  @NotBlank String customerId,
  @NotEmpty
  List<OrderLineRequest> lines
) {}
Response Model
Presentation layer outbound
record OrderResponse(
  Long id,
  String status,
  BigDecimal total
) {}
DTO
Between service / application layers
record OrderSummaryDto(
  Long id,
  String status,
  BigDecimal total
) {}
Entity
Repository layer only
@Entity
class Order {
  @Id Long id;
  String status;
  BigDecimal total;
}
Problems caused by using the entity as an API type

Schema becomes the API contract

Add a column, it appears in the JSON. Rename a field, a client breaks. The database and the API are independently deployable concerns — they must not be the same object.

Jackson and lazy loading

Serialising a JPA entity outside a transaction triggers LazyInitializationException on any lazily loaded association. The temptation to set everything to EAGER follows, and performance degrades silently.

Mass assignment risk

// Client can POST an id, override internal fields
@PostMapping
public void create(@RequestBody Order order) {
    orderRepository.save(order); // dangerous
}

Mapping with MapStruct

Eliminates boilerplate without runtime reflection cost.

@Mapper(componentModel = "spring")
public interface OrderMapper {
    OrderSummaryDto  toDto(Order entity);
    Order            toEntity(PlaceOrderDto dto);
    OrderResponse    toResponse(OrderSummaryDto dto);
}
// Section 04

Mappers

Mappers are the mechanism that makes boundary discipline enforceable in practice. Without them, translation logic bleeds into services and controllers, coupling them to the shape of types they should not know about.

Each boundary owns exactly one mapper. A change on either side of that boundary is contained to the mapper — nothing else needs to change.

ControllerPlaceOrderRequest → PlaceOrderDto
↓  [Request Mapper]  ↓
ServicePlaceOrderDto → Order entity
↓  [Domain Mapper]  ↓
RepositoryOrder entity → OrderSummaryDto
↑  [Domain Mapper]  ↑
ServiceOrderSummaryDto → OrderResponse
↑  [Response Mapper]  ↑
Controller

Without a Mapper

The service is coupled to the internal structure of the entity. A renamed field, changed type, or restructured relationship ripples into service logic.

// Service knows entity internals — every field, every type
public OrderSummaryDto getOrder(Long id) {
    Order entity = orderRepository.findById(id).orElseThrow();
    return new OrderSummaryDto(
        entity.getId(),
        entity.getStatus().name(),       // knows it's an enum
        entity.getTotal(),
        entity.getFirstName() + " " + entity.getLastName() // knows structure
    );
}

With a Mapper

// Service knows nothing of entity structure
public OrderSummaryDto getOrder(Long id) {
    Order entity = orderRepository.findById(id).orElseThrow();
    return orderMapper.toDto(entity);   // one line, all knowledge isolated
}

// All translation knowledge lives here and nowhere else
@Mapper(componentModel = "spring")
public interface OrderMapper {
    @Mapping(target = "fullName",
             expression = "java(e.getFirstName() + ' ' + e.getLastName())")
    OrderSummaryDto  toDto(Order e);
    Order            toEntity(PlaceOrderDto dto);
    OrderResponse    toResponse(OrderSummaryDto dto);
}
What coupling risk does a mapper specifically reduce?

Schema changes contained to one place

Adding a deletedAt soft-delete column, changing status from String to an enum, or splitting name into firstName/lastName — without a mapper these changes propagate into service methods. With one, they're contained to the mapper and the entity.

DTO evolution independent of entity evolution

The API might need to expose computed or combined fields the entity doesn't naturally surface. The mapper absorbs that transformation without polluting the service.

MapStruct vs manual mapping

ApproachProsCons
MapStructCompile-time, zero reflection, IDE navigableAnnotation processor setup required
Manual in serviceNo dependencyCouples service to entity; scattered across methods
Mapper class (manual)Centralised, no annotation processorVerbose; easy to miss fields

MapStruct is the pragmatic default. It generates readable Java at compile time, fails fast on unmapped fields, and has no runtime cost.

// Section 05

CQRS in the Service Layer

Command Query Responsibility Segregation at its simplest is one rule: methods that change state return nothing, methods that read state change nothing. No buses, no event sourcing, no second database required.

@Service
public class OrderService {

    // ── Commands: void, @Transactional ───────────────────────────
    @Transactional
    public void placeOrder(PlaceOrderRequest req) {
        Order order = Order.place(req.customerId(), req.lines());
        orderRepository.save(order);
    }

    @Transactional
    public void shipOrder(Long id) {
        Order order = orderRepository.findById(id).orElseThrow();
        order.ship();
        orderRepository.save(order);
    }

    // ── Queries: return data, readOnly = true ────────────────────
    @Transactional(readOnly = true)
    public OrderResponse getOrder(Long id) {
        return orderRepository.findById(id)
            .map(this::toResponse)
            .orElseThrow();
    }

    @Transactional(readOnly = true)
    public List<OrderSummaryResponse> getByCustomer(String customerId) {
        return orderRepository.findByCustomerId(customerId)
            .stream().map(this::toSummary).toList();
    }
}
Benefits of the command/query split

Commands return void

Stops the habit of saving an entity and immediately returning it as the API response — which is usually the entity leaking out disguised as a response model. If the client needs to know the new ID, return just the ID, not the entity.

Read-only transactions

Hibernate skips dirty checking for readOnly = true transactions. On a busy read path this has a measurable impact. It also documents intent.

Queries can bypass the domain model

If a query becomes a performance concern, it can go straight to a JPA projection without touching domain logic. The seam is already there.

@Transactional(readOnly = true)
public List<OrderSummaryResponse> getByCustomer(String id) {
    // Projection — no entity hydration, no domain model
    return orderJpaRepository.findSummariesByCustomerId(id);
}

// In your Spring Data repo
interface OrderJpaRepository extends JpaRepository<Order, Long> {
    @Query("SELECT new com.example.OrderSummaryResponse(o.id, o.status, o.total)" +
           " FROM Order o WHERE o.customerId = :id")
    List<OrderSummaryResponse> findSummariesByCustomerId(String id);
}

Thin Controllers

A controller that makes business decisions is a service in disguise. Keep controllers as a thin translation layer between HTTP and your application.

@RestController
@RequestMapping("/orders")
public class OrderController {

    private final OrderService orderService;

    // ── Command endpoint ─────────────────────────────────────────
    @PostMapping
    public ResponseEntity<Void> placeOrder(
            @Valid @RequestBody PlaceOrderRequest req) {
        orderService.placeOrder(req);
        return ResponseEntity.accepted().build();
    }

    // ── Query endpoint ───────────────────────────────────────────
    @GetMapping("/{id}")
    public OrderResponse getOrder(@PathVariable Long id) {
        return orderService.getOrder(id);
    }
}

Controller Packages

Where a controller lives in the package structure is an organisational decision entirely independent of what it is allowed to call. The two concerns should not be conflated.

Placing a controller inside a feature package implies the controller belongs to that domain. But a controller that orchestrates across multiple bounded contexts belongs to none of them — it is a delivery mechanism concern sitting in front of the application.

Feature Package

Controllers Package
Separate delivery layer

  OrderController.java ← here

A controllers package becomes the more honest choice when controllers are orchestrating across context boundaries:

// controllers/CheckoutController.java
// Orchestrates across three bounded contexts —
// does not belong inside any single one
@RestController
@RequestMapping("/orders")
public class CheckoutController {

    private final OrderService     orderService;      // orders/
    private final PaymentService   paymentService;    // payments/
    private final NotificationService notificationService; // notifications/

    @PostMapping("/{id}/checkout")
    public CheckoutResponse checkout(@PathVariable Long id) {
        Order         order  = orderService.getOrder(id);
        PaymentResult result = paymentService.charge(order);
        notificationService.confirmOrder(order, result);
        return new CheckoutResponse(order, result);
    }
}
Cross-context calls: controller vs service

The controller is one of the few legitimate places to orchestrate across bounded context boundaries. A service calling another context's service is a warning sign — it creates direct coupling between contexts that domain events are designed to avoid.

CallerCross-context callVerdict
ControllerCalls OrderService + PaymentService✓ Acceptable — delivery mechanism orchestration
OrderServiceCalls PaymentService directly⚠ Caution — tight context coupling; prefer domain events
Order aggregateCalls PaymentService directly✗ Never — domain must not know infrastructure

Coding to Interfaces

The value of coding to interfaces in a modern Spring Boot project is more nuanced than convention suggests. Blanket interface-per-service adds indirection without a meaningful return in most cases.

Modern reality: Mockito mocks concrete classes. CGLIB proxies concrete classes for @Transactional and AOP. IntelliJ refactors concrete classes trivially. The mechanical safety net interfaces once provided is largely covered by tooling.

Where Interfaces Earn Their Place

// ✓ Repository port — genuine alternative implementations exist
public interface OrderRepository {
    Optional<Order> findById(Long id);
    void save(Order order);
}

@Repository
class JpaOrderRepository implements OrderRepository { ... }  // production

class InMemoryOrderRepository implements OrderRepository { ... }  // tests


// ✓ External service integration — you don't own the other side
public interface PaymentGateway {
    PaymentResult charge(Money amount, String reference);
}

class StripePaymentGateway implements PaymentGateway { ... }  // production
class StubPaymentGateway  implements PaymentGateway { ... }   // tests / sandbox


// ✓ Multiple real implementations exist or are planned
public interface NotificationService { ... }
class EmailNotificationService implements NotificationService { ... }
class SmsNotificationService   implements NotificationService { ... }

Where Interfaces Are Noise

// ✗ One implementation, no swap planned, Mockito works without it
public interface OrderService { ... }

@Service
public class OrderServiceImpl implements OrderService { ... }

// Mockito mocks this without an interface
@MockBean OrderService orderService;
Decision guide
ScenarioInterface?Reason
Repository you might swap✓ YesIn-memory impl for tests; different DB for environments
External service (payment, email, SMS)✓ YesStub in tests; real adapter in production
Multiple real implementations✓ YesSpring selects by qualifier or condition
Internal application service, one impl✗ NoMockito + CGLIB make this unnecessary
Service that is @Transactional✗ NoCGLIB proxies concrete classes; interface not required

The interface belongs at the boundary — where something external, swappable, or independently testable lives. Between your own services, inside a single bounded context, it is usually indirection without benefit.

Migrating Towards DDD

DDD is a modelling discipline, not an architectural pattern. The three-tier structure with good discipline already provides the scaffolding. Migration is incremental — you don't need a big bang rewrite.

Key insight: When you push business rules into the entity and stop using public setters for state transitions, you are already doing DDD. The only thing missing is the vocabulary.

Push Rules Into the Entity

The entity defends its own state. Replace public setters with intention-revealing methods. This is the single most impactful step.

// Before
order.setStatus("SHIPPED"); // called from anywhere, no validation

// After
public void ship() {
    if (status != OrderStatus.PENDING) {
        throw new IllegalStateException(
            "Cannot ship order in status: " + status);
    }
    this.status = OrderStatus.SHIPPED;
}

Introduce Value Objects

Replace primitives where the type carries meaning or needs validation. Java records make this near-zero cost.

public record CustomerId(String value) {
    public CustomerId {
        if (value == null || value.isBlank())
            throw new IllegalArgumentException("CustomerId cannot be blank");
    }
}

public record Money(BigDecimal amount, Currency currency) {
    public Money add(Money other) {
        if (!this.currency.equals(other.currency))
            throw new IllegalArgumentException("Currency mismatch");
        return new Money(this.amount.add(other.amount), this.currency);
    }
}

Validation moves into the type. Services stop repeating the same null checks and format guards.

Name the Aggregate Root

Your entity that protects state is already an aggregate. Acknowledge it — and enforce that child entities (e.g. OrderLine) are only accessed through the root.

@Entity
public class Order { // Aggregate root

    @OneToMany(cascade = CascadeType.ALL, orphanRemoval = true)
    private List<OrderLine> lines = new ArrayList<>();

    // Only the root adds/removes lines
    public void addLine(OrderLine line) {
        if (status != OrderStatus.PENDING)
            throw new IllegalStateException("Order is not pending");
        lines.add(line);
    }
}

// OrderLine has no public repository of its own
// Nothing calls orderLineRepository.save() directly

Introduce Domain Events

Aggregates signal meaningful state changes. Other parts of the system react without the aggregate knowing who is listening.

// Domain event — a plain record
public record OrderShipped(Long orderId, Instant occurredAt) {}

// Aggregate raises the event
public void ship() {
    if (status != OrderStatus.PENDING)
        throw new IllegalStateException("Cannot ship");
    this.status = OrderStatus.SHIPPED;
    registerEvent(new OrderShipped(this.id, Instant.now()));
}

// Service publishes after saving
@Transactional
public void shipOrder(Long id) {
    Order order = orderRepository.findById(id).orElseThrow();
    order.ship();
    orderRepository.save(order);
    order.getDomainEvents().forEach(eventPublisher::publishEvent);
}

Organise Into Bounded Contexts

Reorganise packages around business capabilities. Each context is self-contained. Contexts communicate through domain events, not direct service calls.

com.example.myapp
├── controllers/                  ← delivery mechanism; orchestrates across contexts
│   ├── OrderController.java
│   └── CheckoutController.java
│
├── orders/                       ← bounded context
│   ├── Order.java                (aggregate)
│   ├── OrderLine.java            (child entity)
│   ├── OrderStatus.java
│   ├── CustomerId.java           (value object)
│   ├── Money.java                (value object)
│   ├── OrderShipped.java         (domain event)
│   ├── OrderRepository.java
│   └── OrderService.java
│
└── shipping/                     ← another bounded context
    ├── Shipment.java
    ├── ShipmentService.java
    └── OrderShippedHandler.java  ← listens to event, no OrderService import
Rule: shipping never imports from orders except for event types in a shared events package. Contexts are decoupled by design.

Comparison: Three-Tier vs Clean Architecture

Three-Tier architecture, properly structured, is an excellent and complete solution for the majority of applications. Clean Architecture is not a superior replacement — it solves a specific set of problems that most medium-sized applications do not have. The discipline described in this guide closes most of the apparent gaps between the two approaches.

Concern Three-Tier (Disciplined) Clean / Hexagonal Architecture
Layer structure Controller → Service → Repository Presentation → Application → Domain → Infrastructure
Dependency direction ✓ Downward only (enforced by convention) ✓ Inward only (enforced by ArchUnit / package structure)
Repository abstraction Spring Data interface is the abstraction Domain-owned port interface; JPA is the adapter
Domain model JPA entity can carry domain logic Domain model is independent of JPA; mapped separately
Infrastructure swap ✓ Easy — with port interfaces (as recommended above) ✓ Easy — swap the adapter, port unchanged
Testability (domain) ✓ Entities unit-test cleanly; JPA annotations are passive metadata ✓ Pure domain objects, zero framework dependency
Multiple delivery mechanisms ⚠ Requires care — uncommon in practice for most applications ✓ Use cases are delivery-agnostic by design
Boilerplate ✓ Low — faster initial delivery, lower ongoing maintenance cost ✗ Higher — ports, adapters, mappers, use case classes, separate entities; cost compounds with scale
Team onboarding ✓ Familiar to most Spring developers ⚠ Steeper initial learning curve
CQRS support ✓ Natural fit in the service layer ✓ Explicit use case classes per command/query
DDD compatibility ✓ Yes — domain logic in entity, value objects, events ✓ Yes — domain layer is purpose-built for it
ArchUnit enforcement ✓ Downward-dependency rule is straightforward to enforce ✓ Rules map directly to explicit layer packages
Best for Most teams and most applications. Correct default choice up to large scale. Complex domains with JPA-hostile models; large teams needing enforced hard boundaries; applications with multiple delivery mechanisms.
The default choice should be Three-Tier. Clean Architecture is not a better version of the same thing — it is a different trade-off suited to a narrower set of problems. For most medium-sized applications, the disciplined Three-Tier structure described in this guide — with port interfaces at persistence and external service boundaries — is a complete, maintainable, and production-proven architectural end state. Choosing Clean Architecture for an application that does not need it adds cost without benefit.

// Section 12

Ports Without Full Clean Architecture

Extracting a repository port interface gives you the most valuable part of hexagonal architecture without the overhead of the full pattern. For most medium-sized applications, this is a stable end state — not a waypoint to something more complex.

Key insight: When the service has no compile-time dependency on JPA and persistence is swappable, you have already gained the concrete day-to-day benefit of hexagonal architecture. The rest solves problems most medium-sized applications do not have.

The Pragmatic Structure

com.example.myapp
├── controllers/
│
├── orders/
│   ├── Order.java                ← JPA entity carrying domain logic
│   ├── OrderRepository.java      ← port interface, domain-owned
│   ├── OrderService.java         ← depends on the port, not JPA
│   └── infrastructure/
│       └── JpaOrderRepository.java  ← implements the port, hidden here
│
└── payments/
    ├── Payment.java
    ├── PaymentGateway.java       ← port for external service
    ├── PaymentService.java
    └── infrastructure/
        └── StripePaymentGateway.java

The service depends on OrderRepository the interface. It has no knowledge of JPA, Hibernate, or the database. The infrastructure subfolder is an implementation detail — nothing outside it needs to change if the implementation changes.

// Service depends only on the port
@Service
public class OrderService {
    private final OrderRepository orderRepository; // interface, not JPA

    @Transactional
    public void shipOrder(Long id) {
        Order order = orderRepository.findById(id).orElseThrow();
        order.ship();
        orderRepository.save(order);
    }
}

// Port — owned by the domain, knows nothing of JPA
public interface OrderRepository {
    Optional<Order> findById(Long id);
    void save(Order order);
}

// Adapter — hidden in infrastructure, the only JPA-aware class
@Repository
class JpaOrderRepository implements OrderRepository {
    private final OrderJpaRepository jpa; // Spring Data

    @Override
    public Optional<Order> findById(Long id) {
        return jpa.findById(id);
    }

    @Override
    public void save(Order order) {
        jpa.save(order);
    }
}

The Cost-Benefit of Going Further

Step Benefit Cost Worth it?
Repository port interface Swappable persistence, testable without DB One extra interface per repository ✓ Almost always
External service port (PaymentGateway etc.) Stubbable, swappable provider One extra interface per integration ✓ Almost always

When does full clean architecture justify itself?

The crossover point is usually one or more of the following conditions being true simultaneously:

ConditionWhy It Changes the Calculation
Domain model fights JPA annotationsComplex inheritance, non-standard mapping, or persistence strategies that distort the domain — separate entities become necessary rather than optional
Multiple persistence backends in one applicationPorts become the only clean abstraction over heterogeneous storage
Same use case invoked from REST, messaging, and scheduled jobsA delivery-agnostic use case class pays for itself when there are three callers
Team large enough that convention breaks downEnforced package boundaries via ArchUnit prevent coupling that discipline alone cannot

For most medium-sized applications none of these apply. The disciplined three-tier structure with ports at the persistence and external service boundaries is a maintainable end state in its own right.

// Section 13

When to Evolve Beyond Three-Tier

Adding architectural complexity before the problem demands it is a cost with no benefit. These are signals that the disciplined three-tier structure is approaching its limits.

Signal What It Suggests
Services are God classes Use case slicing into command/query handlers; consider bounded contexts
Need to swap persistence (e.g. Postgres → Mongo) Extract domain-owned repository port; move JPA to adapter
Same logic served over REST, messaging, and scheduled jobs Application use case layer independent of delivery mechanism
Large team stepping on each other Explicit bounded contexts with enforced ArchUnit boundaries
Domain model needs to be tested without JPA Separate domain objects from JPA entities; map between them
Business rules spread across services Rich aggregates; domain events; push logic into the model
Guiding principle: The disciplined Three-Tier structure described in this guide — distinct types, logic in the model, CQRS split, port interfaces at boundaries — is a legitimate architectural end state for most applications, not a waypoint to something more complex. If you do eventually need to go further, the seams already exist and migration is straightforward. If you don't apply this discipline, no amount of architectural ambition will rescue a codebase where entities leak everywhere and services do everything.
