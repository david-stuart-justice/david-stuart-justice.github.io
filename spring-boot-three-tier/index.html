Spring Boot Architecture — Where to Start

The Spring Initializer gets you running in minutes. It does not give you an architecture. These are the principles that make the default structure sustainable as the codebase grows.

Spring Boot Java 21+ Three-Tier Architecture
The Starting Point

What Spring Boot Gives You — and What It Doesn't

Spring Boot's scaffold is a reasonable starting point: a @RestController, a @Service, a JPA repository, and an entity. Requests arrive, data is saved, responses are returned. It works.

What it doesn't provide is discipline around where logic lives, which types cross which boundaries, or what happens when business rules change. Without explicit rules, the scaffold drifts:

  • Entities travel unchanged from repositories to controllers, coupling the API contract directly to the database schema. A column rename breaks a client.
  • Business rules accumulate in services until they become large coordination classes that are difficult to test and expensive to change.
  • Tests require a full Spring context because the service depends directly on JPA, making unit tests slow or impossible.
  • A lazy-loading exception appears in production when Jackson tries to serialise an entity outside a transaction.
  • Two developers editing related features break each other's work because there are no enforced boundaries between concerns.

None of this is caused by Spring Boot. It is caused by the absence of enforced boundaries.

The Structure

The Three-Tier Model

The default scaffold already maps to a three-tier structure. The discipline is in enforcing what each tier is allowed to do and what types may cross each boundary.

The direction of dependency is fixed: downward only. Controllers know about services. Services know about repositories. Nothing calls back up through the layers. This is the rule that everything else depends on.

Core Principles

Three Principles That Prevent Drift

Principle 01
Strict Layer Boundaries
Each layer owns one responsibility. Controllers own HTTP. Services own business logic. Repositories own persistence. No layer reaches past the next one.
Without this: business logic accumulates in controllers and controllers call repositories directly. The service layer becomes optional, then absent.
Principle 02
Distinct Types at Every Boundary
Each layer owns its own types. The API request is not the service DTO. The service DTO is not the JPA entity. Mappers translate at each crossing.
Without this: a renamed column breaks an API client. A schema change propagates through every layer. The database and the API become the same thing.
Principle 03
Business Logic Belongs in the Domain Model
State transitions belong on the entity. order.ship() belongs on Order, not duplicated across every service that handles orders.
Without this: the same rule is written in multiple places. One copy is updated; others are not. Tests target service methods when they should target domain behaviour.
Why It Works

The Principles Create Seams

Each principle creates a seam — a boundary at which one part of the codebase can change without affecting another. The seams are what make a codebase maintainable rather than fragile.

  • Layer boundaries mean an HTTP API change — a new header, a different status code, a revised serialisation format — does not reach business logic.
  • Type boundaries mean a schema change — a renamed column, a split field, a new table — does not reach controllers. The mapper absorbs it.
  • Domain logic on entities means business rules have one home. They are tested once, applied consistently, and found in one place when they need to change.

The seams do not appear automatically. They require a deliberate decision at every layer boundary to translate types rather than pass them through, and a deliberate decision on every entity to protect state rather than expose setters.

The benefit compounds over time. A codebase built on these principles in its first six months is straightforward to extend at eighteen months. One that skipped the discipline is usually being rewritten.
The Bigger Picture

Three-Tier Is the Right Default — Not a Starting Point

Disciplined Three-Tier is a complete and maintainable architectural end state for most Spring Boot applications. It is not a stepping stone to something more sophisticated.

Default choice
Disciplined Three-Tier
Layer boundaries, type discipline, domain logic on entities, port interfaces at persistence boundaries. Correct for most teams and most applications.
When useful
+ Selective DDD
Value objects, aggregate roots, domain events. Add where the domain model benefits — not as a blanket policy.
Specific problems only
+ Clean / Hexagonal
Separate domain objects, explicit adapter packages, delivery-agnostic use cases. Justified by JPA-hostile models, multiple delivery mechanisms, or large teams needing enforced hard boundaries.

Each step adds cost — more classes, more abstraction, steeper onboarding. The right decision is to apply the discipline in this guide first and move further only when a specific problem demands it. Migration is straightforward when the seams are already in place.
