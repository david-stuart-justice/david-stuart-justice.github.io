Core Concepts

Pattern Independence

Event sourcing, event streaming, and event-driven architecture are three independent patterns that are often confused. You can use any one without the others. This guide clarifies what each pattern does and when to combine them.

Critical Insight

The Three Patterns Are Independent

The word "event" appears in all three patterns, leading to confusion. But they solve different problems and can be used independently.

Event Sourcing

Problem: How to persist state

Solution: Store state as events, rebuild by replay

Scope: Single service, internal

Event Streaming

Problem: How to move data between services

Solution: Durable message brokers (Kafka, Kinesis)

Scope: Cross-service integration

Event-Driven Architecture

Problem: How services communicate

Solution: Asynchronous events instead of synchronous calls

Scope: System design philosophy

Common Combinations

Pattern 1

Event Sourcing Alone

Use event sourcing for persistence and CQRS within a single service. No message broker needed. Expose data via REST/GraphQL APIs.

Command → Event Store → Read Models → API
                                   ↓
                            External Consumers Query

Benefits

  • Complete audit trail
  • Time travel queries
  • Rebuild read models from history
  • Add new projections without data migration

Trade-offs

  • Consumers coupled to your API
  • API is single point of failure
  • Synchronous dependencies
When to use: You need event sourcing benefits (audit, replay, time travel) but don't need asynchronous cross-service integration.
Pattern 2

Event Streaming Alone (CRUD + Kafka)

Traditional database with row updates. Publish thin events to Kafka for integration, analytics, and notifications.

Command → UPDATE database → Publish event
                                   ↓
                     Kafka → External Consumers

Benefits

  • Loose coupling between services
  • Asynchronous, resilient integration
  • Consumers build custom projections
  • Simpler than event sourcing

Trade-offs

  • No historical replay (unless Kafka retention is very long)
  • No built-in audit trail
  • Cannot rebuild state from events
When to use: Most microservices. You need asynchronous integration but don't need event sourcing's time travel and audit capabilities.
Pattern 3

Event Sourcing + Event Streaming

Full combination: event sourcing for persistence, publish events to Kafka for external integration.

Command → Event Store (rich events)
            ↓
            Internal Projections
            ↓
            Transform to thin events
            ↓
            Kafka → External Consumers

Benefits

  • All event sourcing benefits (audit, replay)
  • Plus loose coupling with external services
  • Rich events internally, thin events externally

Trade-offs

  • Most complex approach
  • Two infrastructure pieces (event store + Kafka)
  • Event transformation layer needed
When to use: You need both event sourcing benefits AND asynchronous cross-service integration. High-value domains with complex audit requirements.
Pattern 4

Traditional CRUD (No Events)

Standard database with synchronous REST/GraphQL APIs. No event sourcing, no event streaming.

Command → UPDATE database → REST API
                                   ↓
                         Consumers Query API

Benefits

  • Simplest approach
  • Well-understood patterns
  • Fast queries (no event replay)
  • Immediate consistency

Trade-offs

  • No audit trail (unless added separately)
  • No time travel queries
  • Tight coupling between services
  • Synchronous dependencies
When to use: Most applications. Use this unless you have specific requirements for audit trails, time travel, or asynchronous integration.
Key Distinctions

Rich Events vs Thin Events

This is a critical but often overlooked distinction. Events used for different purposes have different content.

Rich Events (Event Sourcing)

Purpose: Rebuild aggregate state

Storage: Event store (PostgreSQL table)

Audience: Internal to the service

Content: Everything needed to reconstruct state

Retention: Forever (source of truth)

Example

ItemAddedEvent {
  sku: "SKU-123",
  quantity: 2,
  price: 29.99,
  currency: "GBP",
  variantDetails: {...},
  taxCalculation: {...},
  discounts: [...],
  inventoryReservationId: "...",
  warehouseLocation: "..."
}

Thin Events (Event Streaming)

Purpose: Notify consumers something happened

Storage: Message broker (Kafka topic)

Audience: External services

Content: Just enough to react or query for details

Retention: Days to weeks

Example

OrderConfirmedEvent {
  orderId: "ORD-1001",
  customerId: "CUS-8821",
  totalAmount: 125.50,
  currency: "GBP"
}

// Consumers query Order API
// if they need full details

Hybrid approach: Use rich events internally for event sourcing. Transform to thin events when publishing to external message brokers. This gives you both internal flexibility and external loose coupling.

Decision Framework

Choosing Your Approach

Start with these questions:

  1. Do you need a complete audit trail? → Consider event sourcing
  2. Do you need time travel queries? ("What was state on March 15th?") → Consider event sourcing
  3. Do you need to rebuild state from history? → Consider event sourcing
  4. Do external services need to react to changes? → Consider event streaming
  5. Do you need asynchronous, loose coupling? → Consider event streaming
  6. Do consumers need custom projections? → Consider event streaming

If you answered "yes" to 1-3 but "no" to 4-6:

Event Sourcing + APIs

Get audit and replay benefits without message broker complexity.

If you answered "no" to 1-3 but "yes" to 4-6:

CRUD + Event Streaming

Traditional database with Kafka for integration. Most common pattern.

If you answered "yes" to all:

Event Sourcing + Event Streaming

Full combination. Most complex but most powerful for high-value domains.

If you answered "no" to all:

Traditional CRUD

Keep it simple. This works for most applications.

Common Misconceptions

What People Get Wrong

❌ Myth: Event sourcing requires Kafka/Kinesis

Reality: Event sourcing is a persistence pattern. It works entirely within one service using a database table as the event store. Message brokers are optional, only needed for cross-service integration.

❌ Myth: Event streaming requires event sourcing

Reality: Most event-driven systems use traditional CRUD databases and publish change events to Kafka. Event sourcing is not required.

❌ Myth: Event sourcing events and streaming events are the same

Reality: Sourcing events are rich (rebuild state). Streaming events are thin (notifications). They serve different purposes and usually have different schemas.

❌ Myth: Read-side projections require message brokers

Reality: Internal projections read directly from the event store. Message brokers are only for external consumers in different services.
