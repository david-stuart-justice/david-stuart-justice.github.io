Conceptual Overview

Event Patterns: Streaming, Sourcing, and Event-Driven Architecture

Event streaming, event sourcing, and event-driven architecture are distinct patterns often confused. This guide defines each, explains when to use them, and clarifies common misconceptions. No implementation details—pure concepts.

Pattern Comparison

Event Streaming vs Event Sourcing vs Event-Driven Architecture

Pattern Purpose Key Characteristic
Event Streaming Data transport mechanism Durable logs for moving events between services at scale
Event Sourcing Persistence pattern Store state as a sequence of events; rebuild current state by replay
Event-Driven Architecture System design style Services communicate primarily through asynchronous events

These patterns are independent. You can use event streaming without event sourcing (most event-driven systems). You can use event sourcing without event streaming (purely internal persistence + CQRS). Many systems combine both when they need event sourcing for persistence AND cross-service integration.

Critical Distinction

Event Content: Sourcing Events vs Streaming Events

Events used in event sourcing to rebuild state are often richer and more complete than events published to integration streams. This is a common point of confusion.

Event Sourcing Events

Purpose: Reconstruct aggregate state by replay

  • Must contain all data needed to rebuild the aggregate
  • Often richer with full domain details
  • Scoped to one aggregate's lifecycle
  • Retained forever (source of truth)
  • Example: ItemAdded includes sku, quantity, price, variant details, tax calculation, discounts—everything to rebuild Order state

Event Streaming Events

Purpose: Notify consumers that something happened

  • Contain just enough data for consumers to react
  • Often thinner, notification-focused
  • Cross-service integration
  • Shorter retention (days/weeks)
  • Example: OrderConfirmed includes orderId, customerId, total—consumers query Order API if they need full details

In Practice

  • Event-sourced systems often publish their domain events to a stream, but may strip internal state details, enrich with context, or transform into different event types for different bounded contexts.
  • Streaming-first systems (state in database, events for integration) publish thinner events because the source of truth is the database, not the event log.
  • Hybrid approach: Use rich events internally for event sourcing, publish derived notification events to external streams for integration.
Do not assume event sourcing events and integration events have the same schema. They serve different purposes and often need different content.
