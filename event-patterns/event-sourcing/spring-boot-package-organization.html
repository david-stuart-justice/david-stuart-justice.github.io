Architecture Flow

Commands change aggregate state. Aggregates append immutable events to the event store. Projections consume events to build query views. Integration events are published from an outbox worker to avoid dual writes.

Stream and Event Organization

1. Stream Strategy

  • One stream per aggregate instance.
  • Use stream IDs such as order-12345.
  • Keep ordering guarantees inside aggregate boundaries.
  • Avoid one global stream for all events.

2. Event Envelope

  • Metadata fields: eventId, eventType, eventVersion, occurredAt.
  • Traceability: aggregateId, causationId, correlationId.
  • Payload contains only business facts.
  • Events are immutable after append.

3. Event Evolution

  • Prefer additive changes to event payloads.
  • Introduce explicit version numbers.
  • Use upcasters during replay for old formats.
  • Validate schema compatibility in CI.

Adjusting from Spring Initializr Default Package

Spring Initializr usually starts with a compact structure. For event sourcing, split code by responsibility so domain logic stays isolated from transport, persistence, and projection concerns. This guide stays Java and Spring Boot specific.

Initializr Default Recommended Target Reason
service application + domain Separates orchestration from business invariants.
repository infrastructure.eventsourcing Moves from CRUD persistence to append-only stream access.
controller api Keeps transport concerns separate from command handling.
none projection Creates dedicated read model handlers and replay capability.
none infrastructure.messaging Supports outbox publishing, retries, and DLQ handling.

Minimum Interfaces to Define Early

Core Contracts

  • EventStore.append(streamId, expectedVersion, events)
  • EventStore.load(streamId)
  • SnapshotRepository.save/load
  • Projector.handle(eventEnvelope)

Operational Controls

  • Deduplicate by eventId for idempotent consumers.
  • Track replay checkpoints per projector.
  • Expose projection lag and outbox lag metrics.
  • Add CI checks for schema compatibility.
Keep the domain package free from framework dependencies where practical. Spring annotations are fine in application and infrastructure layers, but business rules remain easier to test and evolve when kept in plain Java classes.
Avoid publishing directly to message brokers inside command transactions. Use the outbox pattern so event persistence and publication remain consistent during failures.
