Core Concept

What Event Sourcing Is

Event sourcing stores state as a sequence of immutable events instead of updating current state in place. The event log becomes the source of truth; current state is derived by replaying events.

In a traditional CRUD system, you UPDATE a row when an order is confirmed. The previous state is overwritten. In an event-sourced system, you APPEND an OrderConfirmed event to the log. The current confirmed state is rebuilt by applying all events in order.

Traditional State Storage

  • Current state lives in database rows.
  • UPDATE statements overwrite old values.
  • History requires separate audit tables or versioning.
  • Fast reads via indexed queries.

Event-Sourced Storage

  • State changes are append-only events.
  • No UPDATE or DELETE, only INSERT.
  • Full history is intrinsic to the model.
  • Reads require replaying events (slow without CQRS).
Decision Framework

When Event Sourcing Fits

Event sourcing is not a universal default. It adds significant complexity. Apply it when the benefits outweigh the cost.

Strong Fit Neutral or Weak Fit
Audit trail is legally or financially mandated Audit is nice-to-have or can use database versioning
Temporal queries are frequent ("state as of date X") Temporal queries are rare or one-off reports
Domain has clear aggregates with well-defined invariants Domain is mostly CRUD with simple validation
Event history has business value (fraud analysis, ML training) Only current state matters for operations
Aggregate event count manageable (hundreds, not millions) High-frequency writes per aggregate make replay prohibitive
Multiple downstream systems need independent views Single service, no integration needs
If the only goal is integration or pub/sub, event streaming without event sourcing is simpler. Save state in a normal database and publish change events to a stream.
Essential Pattern

CQRS: Command Query Responsibility Segregation

CQRS separates write models from read models. Commands update aggregates; queries hit read-optimized projections. In event sourcing, CQRS is almost always necessary because querying the event store directly is too slow.

Write Side (Command)

  • Aggregates enforce business rules.
  • Commands validated against invariants.
  • Events appended to event store on success.
  • Optimized for consistency and correctness.

Read Side (Query)

  • Projections are denormalized views.
  • Built by consuming events asynchronously.
  • Optimized for specific query patterns.
  • Eventually consistent with write side.

One aggregate can feed multiple read models. One read model can combine events from multiple aggregates. Read models are disposable—if corrupted, rebuild from events.

Consistency Boundary

Aggregates and Consistency

An aggregate is a consistency boundary—a cluster of objects treated as a single unit for write operations. In event sourcing, each aggregate instance has its own sequence of events in the event store.

Aggregate Rules

  • One event sequence per aggregate instance. Order aggregate ORD-1001 has its own sequence separate from ORD-1002.
  • Commands enforce invariants before emitting events. Invalid state is never persisted.
  • Events within one aggregate are totally ordered. Sequence numbers provide clear ordering.
  • Cross-aggregate rules use sagas or process managers. Aggregates do not call other aggregates directly.

Example: Order Aggregate

Commands: CreateOrder, AddItem, RemoveItem, ConfirmOrder, ShipOrder, CancelOrder.

Events: OrderCreated, ItemAdded, ItemRemoved, OrderConfirmed, OrderShipped, OrderCancelled.

Invariants: Cannot confirm an empty order. Cannot add items after confirmation. Cannot cancel after shipment started.

Inventory availability is a cross-aggregate concern. Check synchronously in the command handler or coordinate via a saga, but do not encode it as an Order aggregate invariant.

Critical Challenge

The Performance Problem

Rebuilding state by replaying hundreds or thousands of events is slow. This is the primary operational cost of event sourcing.

Scenario: An Order aggregate with 500 events (created, items added/removed, status changes, notes, address updates).

  • Traditional read: 1–5ms indexed database query.
  • Event-sourced read: Load 500 events, deserialize each, apply business logic sequentially. 50–500ms or more.

Mitigation Strategies

1. Snapshots

Save aggregate state periodically (every N events). Replay only events since last snapshot.

Adds complexity: snapshot schema evolution, storage overhead, rebuild when format changes.

2. CQRS (Recommended)

Build denormalized read models. Queries hit read models, not the event store.

Most common approach. Queries are fast; writes remain event-sourced for audit.

3. Caching

Cache rebuilt aggregate state in memory. Invalidate on new event append.

Works when read/write ratio is high and working set fits in cache.

Verification Strategy

Testing Event-Sourced Systems

Event sourcing enables clear testing patterns: given a sequence of past events, when a command is applied, then expect specific new events.

Unit Tests: Aggregate Behavior

Given:
  OrderCreated(orderId=1001, customerId=88)
  ItemAdded(sku=WIDGET-A, qty=2)

When:
  ConfirmOrder(orderId=1001, processedBy=USR-5)

Then:
  OrderConfirmed(orderId=1001, total=20.00, confirmedAt=...)

Integration Tests: Event Store Round-Trip

  • Append events to event store.
  • Load aggregate from store.
  • Verify state rebuilt correctly.
  • Validate optimistic concurrency check.

Projection Tests: Read Model Consistency

  • Publish events to projection handler.
  • Assert read model updated correctly.
  • Verify idempotency (duplicate event ID ignored).
  • Confirm full rebuild from offset 0 produces same state.
Reliable Publishing

The Outbox Pattern

Publishing events to a message broker after a database commit creates a dual-write problem: the commit may succeed but the publish may fail, leaving events stranded.

The outbox pattern solves this by writing events to an outbox table in the same transaction as the event store append. A separate process reliably publishes outbox rows to the broker.

Step 1: Atomic Write

Append event to event store and insert row to outbox table in one database transaction.

Step 2: Async Publish

Outbox worker polls for unpublished rows and publishes to message broker.

Step 3: Mark Published

After successful publish, update outbox row status or delete it.

Step 4: Retry on Failure

If publish fails, row remains unpublished and worker retries on next poll.

This guarantees at-least-once delivery. Consumers must deduplicate using event IDs to achieve exactly-once processing semantics.

Cross-Aggregate Workflows

Sagas and Process Managers

Aggregates enforce invariants within one consistency boundary. Workflows spanning multiple aggregates require coordination outside aggregate logic.

A saga (or process manager) listens for events from multiple aggregates, tracks workflow state, and emits commands or integration events to drive the process forward.

Example: Order Fulfillment Saga

Goal: Ship an order only after payment captured and inventory reserved.

  • Listen for: OrderConfirmed, PaymentCaptured, InventoryReserved.
  • Track state in a small saga state table keyed by orderId.
  • When all three events received, emit ShipOrder command.
  • Handle out-of-order events and timeouts gracefully.
Sagas must be idempotent. The same event may be delivered multiple times; the saga should not duplicate actions or corrupt state.
Practical Limits

When Not to Use Event Sourcing

Event sourcing is a specialized pattern. Most systems should not use it. These are clear signals to avoid event sourcing:

  • Domain is mostly CRUD. If there are no complex invariants or audit requirements, traditional state storage is simpler and faster.
  • High write frequency per aggregate. Sensor data, logs, or analytics with millions of events per aggregate make replay prohibitively slow.
  • Strong consistency required across aggregates. Event sourcing makes cross-aggregate transactions difficult. Use a monolith or distributed transactions if needed.
  • Team lacks operational experience. Debugging event replay, handling schema evolution, and managing projection lag require expertise.
  • Deletion or GDPR compliance is critical. Immutable append-only logs make true deletion nearly impossible without complex tombstone or encryption strategies.
  • Query patterns are complex and ad-hoc. Building many different read models for varied queries adds significant maintenance cost.

If you need integration or pub/sub, use event streaming with traditional state storage. Reserve event sourcing for domains where audit, replay, and temporal queries are central requirements.
