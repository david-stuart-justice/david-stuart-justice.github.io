Core Definitions

What These Patterns Mean

Event Sourcing

A persistence pattern where state is stored as an append-only sequence of events instead of updating current state in place. To get current state, replay all events for that entity.

Events are the source of truth. Database rows are derived.

Event Streaming

A data transport mechanism where services publish events to durable ordered logs (like Kafka or Kinesis). Other services consume those events independently to build projections or trigger workflows.

Events move between services. Can be used with or without event sourcing.

Event-Driven Architecture

A system design style where services communicate primarily through asynchronous events rather than synchronous calls. Events represent facts that happened, not commands.

Services are loosely coupled, reacting to events as they occur.

Common confusion: Event streaming and event sourcing are often conflated but serve different purposes. Streaming moves events between services; sourcing stores state as events. You can use streaming without sourcing.

Shared Assumptions

Project Constraints

Scale & Technical Baseline

  • Volume: Moderate transaction volume.
  • Infrastructure: Container orchestration, message broker, PostgreSQL.
  • Pipeline: CI/CD automation, container deployments.
  • Stack: Java 25, Spring Boot 4, Flyway migrations.

Core Objectives

  • Auditability: Full history of state changes is mandatory.
  • Decoupling: Downstream services consume via events.
  • Reliability: At-least-once delivery to message broker.
  • Maintainability: Code must be readable by generalist engineers.
