Implementation Guide

Event Streaming: Practical Patterns and Design Choices

Concrete guidance for designing event payloads, handlers, and integration patterns when using event streaming platforms like Kafka, Pulsar, Kinesis, or Event Hubs.

Foundations

The Basic Flow

A service processes a command, validates it, updates its database, and publishes an event to a stream. Other services subscribe to that stream and react by updating their own models or triggering workflows. 

Command          Write Service       Event Stream        Consumer Services
   |                  |                    |                      |
   |--CreateOrder---->|                    |                      |
   |                  |--validate--------  |                      |
   |                  |--save to DB-----   |                      |
   |                  |--OrderCreated----->|                      |
   |                  |                    |----OrderCreated----->|
   |<----200 OK-------|                    |                      |
   |                  |                    |                      |--update projection
   |                  |                    |                      |--trigger workflow

Events are published after the database commit succeeds. Consumers receive events in partition order and process them independently.

Core Component

Event Stream

An ordered, append-only log partitioned by a key. Kafka, Pulsar, Kinesis, and Event Hubs are common platforms.

  • Ordered per partition key (for example per orderId).
  • Consumers track their offset and can replay from earlier positions.
  • Retention is time-based or size-based; older events may be deleted or archived.
ordered append-only partitioned
Core Component

Handler

Code in a consumer service that processes one event. Handlers are not part of the event payload. They run asynchronously in services that subscribe to the stream.

  • Updates a read model or projection.
  • Triggers workflows, alerts, or external integrations.
  • Must be idempotent because platforms can deliver duplicates.
Events describe what happened. Handlers decide what to do about it.
Design

Event Payload Design

Include immutable facts about what happened. Provide enough context for consumers to act without querying other services for every event. Exclude mutable data, full object graphs, and confidential content. 

{
  "eventId": "7a81d45e-8ef8-4f64-8f2f-eef2dd4ef9db",
  "eventType": "OrderConfirmed",
  "timestamp": "2026-07-01T10:25:00Z",
  "schemaVersion": 1,
  "correlationId": "req-12f4",
  "causationId": "cmd-99d1",
  "payload": {
    "orderId": "ORD-1001",
    "customerId": "CUS-8821",
    "totalAmount": 125.50,
    "currency": "GBP",
    "confirmedAt": "2026-07-01T10:24:55Z"
  }
}

Line items are not included because the Fulfillment service queries the Order API when it needs full details.

Structure

Metadata vs Payload

  • Metadata supports tracing, versioning, routing, and idempotency.
  • Payload carries business facts used by domain and consumers.
  • Broker metadata can add topic, partition, offset, and key.

Design Choices

Design Choice

Command Input vs Event Output

A REST API may accept a large JSON payload. That is command input, not the event payload. Events carry targeted facts needed by downstream consumers.

Do not copy the full command payload into events. This creates tight coupling between producer and all consumers, bloats the stream, and makes schema evolution difficult.
  • Command: full create/update request with validation rules applied synchronously.
  • Event: immutable facts about what changed, published after successful commit.
  • Example: CreateOrder command with 20 line items emits OrderCreated with orderId, customerId, and total only.
Design Choice

Delta Updates

Publish specific change events rather than full object snapshots after every modification.

Prefer business events over generic patch events. Business events (ShippingAddressChanged) preserve domain intent and meaning. Patch events (FieldUpdated) are purely structural and force consumers to interpret low-level changes.
  • Business events: ShippingAddressChanged, PaymentMethodUpdated (recommended).
  • Generic patch events: OrderDetailsPatched (acceptable for low-value fields only).
  • Include version or revision number to detect stale updates.
Design Choice

Canonical State Location

For systems with large domain objects and selective events, keep the canonical latest state in the write database. Events then carry integration facts, not full state.

  • State-first (common): write store is authoritative, events are change notifications.
  • Event-sourced (complex): event store is authoritative, state rebuilt from events.
  • Hybrid: write store for operations, events for workflows and read projections.
Design Choice

Sensitive Data Policy

Events are durable, replicated, and consumed by multiple services. Deletion is difficult or impossible without breaking audit integrity.

  • Exclude confidential data (payment details, passwords, personal identifiers) by default.
  • Use reference IDs instead of full sensitive content.
  • If inclusion is unavoidable, encrypt payload fields and enforce strict access controls.
