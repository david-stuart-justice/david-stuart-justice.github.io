Event Sourcing Architecture Guide

A practical guide for teams that are newer to hexagonal architecture, DDD, and event-driven design. It explains how to structure a Spring Boot codebase when events are the source of truth, and where each pattern adds value.

Spring Boot 4 Java 25 DDD + Hexagonal Event Sourcing + CQRS
Section 01

Overview

Event sourcing changes the center of the architecture. In a CRUD system, the current table state is the source of truth and events are optional. In an event-sourced system, events are the source of truth and current state is derived.

That shift is why teams often combine event sourcing with DDD and hexagonal boundaries. DDD helps define aggregate boundaries and invariants. Hexagonal structure keeps domain logic independent from persistence, messaging, and delivery protocols.

This guide is intended for systems where replay, auditability, and decoupled downstream consumers are core requirements. If those requirements are not present, a disciplined three-tier model is usually lower cost.
Section 02

When This Architecture Fits

Use this as a decision frame before committing to event sourcing.

Signal Interpretation Recommendation
Strict audit trail for every state transition Historical reconstruction is a hard requirement Strong fit for event sourcing
Multiple downstream systems need independent models Read-side decoupling has high value Strong fit for projections + pub/sub
Domain invariants are complex and long-lived Aggregate boundaries matter for correctness Use DDD aggregates with clear stream boundaries
Mostly CRUD workflows, low domain complexity Event infrastructure cost may exceed benefit Prefer disciplined three-tier

Reference Package Structure

API

Request validation, protocol mapping, response composition. No business invariants.

Application

Command/query orchestration and transactional boundaries. Calls domain methods and ports.

Domain

Aggregates, events, value objects, and invariants. No framework annotations beyond what is necessary.

Section 04

Write Side Flow

Keep command handling deterministic and explicit.

1. Receive commandController validates input shape and calls command handler.
2. Load streamApplication loads aggregate events and rebuilds aggregate state.
3. Enforce rulesAggregate validates invariants and emits new domain events.
4. Persist atomicallyAppend events and outbox rows in one transaction.
@Transactional
public void handle(ApproveOrderCommand cmd) {
    EventStream stream = eventStore.load(cmd.orderId());
    OrderAggregate aggregate = OrderAggregate.rehydrate(stream.events());

    List<DomainEvent> newEvents = aggregate.approve(cmd.processedBy(), clock.instant());

    eventStore.append(
        cmd.orderId(),
        stream.expectedVersion(),
        newEvents
    );

    outboxPort.enqueue(newEvents);
}
The expected version check is the main concurrency control. It prevents lost updates by rejecting appends based on stale aggregate versions.
Section 05

Aggregate Rules and Boundaries

An aggregate is a consistency boundary. Keep it small enough to enforce invariants in one stream, but large enough to prevent invalid state transitions.

Rule Reason
One stream per aggregate instance Clear ordering and optimistic concurrency at the right scope
Commands enforce invariants before emitting events Invalid events are never persisted
Cross-aggregate workflows use saga/process manager Avoid distributed transactions in aggregate handlers
Section 06

Event Store and Snapshots

CREATE TABLE event_store (
  id BIGSERIAL PRIMARY KEY,
  aggregate_id VARCHAR(255) NOT NULL,
  aggregate_type VARCHAR(120) NOT NULL,
  sequence_number BIGINT NOT NULL,
  event_id UUID NOT NULL UNIQUE,
  event_type VARCHAR(200) NOT NULL,
  payload JSONB NOT NULL,
  metadata JSONB,
  created_at TIMESTAMP NOT NULL DEFAULT NOW(),
  UNIQUE(aggregate_id, sequence_number)
);

CREATE TABLE snapshot_store (
  id BIGSERIAL PRIMARY KEY,
  aggregate_id VARCHAR(255) NOT NULL,
  aggregate_type VARCHAR(120) NOT NULL,
  sequence_number BIGINT NOT NULL,
  payload JSONB NOT NULL,
  created_at TIMESTAMP NOT NULL DEFAULT NOW(),
  UNIQUE(aggregate_id, sequence_number)
);

Snapshotting is an optimization, not a source of truth. Rebuild logic must still work correctly from events alone.

Section 07

Read Models and Projections

Read models are disposable and rebuildable. Design them for query use cases, not for domain invariants.

Projection Rules

  • Use idempotent handlers keyed by event ID.
  • Persist checkpoint offsets per projection.
  • Support full rebuild from offset 0.
  • Treat projection schema as query-focused, not canonical.

Failure Handling

  • Retry transient errors with backoff.
  • Route poison events to dead-letter queue.
  • Expose projection lag metrics.
  • Provide replay tooling per projection.
Section 08

Outbox and Integration Events

Publish integration events from an outbox table in the same transaction as the event append. This avoids dual-write inconsistency. 

CREATE TABLE outbox (
  id BIGSERIAL PRIMARY KEY,
  event_id UUID NOT NULL,
  topic VARCHAR(255) NOT NULL,
  payload JSONB NOT NULL,
  published BOOLEAN NOT NULL DEFAULT FALSE,
  attempt_count INT NOT NULL DEFAULT 0,
  created_at TIMESTAMP NOT NULL DEFAULT NOW(),
  published_at TIMESTAMP NULL
);

CREATE INDEX idx_outbox_pending ON outbox (published, created_at);
Use distributed locking for scheduled outbox workers in multi-replica deployments to avoid duplicate relay loops.
Section 09

Sagas and Process Coordination

Use sagas when business processes span multiple aggregates or services with asynchronous steps. 

public class ApplicationCompletionSaga {
    private boolean caseworkerApproved;
    private boolean documentsVerified;

    public List<Command> on(CaseworkerApproved event) {
        caseworkerApproved = true;
        return dispatchIfReady(event.applicationId());
    }

    public List<Command> on(DocumentsVerified event) {
        documentsVerified = true;
        return dispatchIfReady(event.applicationId());
    }

    private List<Command> dispatchIfReady(String applicationId) {
        if (caseworkerApproved && documentsVerified) {
            return List.of(new CompleteApplicationCommand(applicationId));
        }
        return List.of();
    }
}
Section 10

Testing Strategy

Test Type Scope Purpose
Aggregate tests Given events -> when command -> expect events Invariant correctness
Projection tests Given event sequence -> expect read model rows Read-model correctness and idempotency
Contract tests Event schema versions Safe event evolution
Replay tests Rebuild from historical events Operational confidence for rebuilds
Section 11

Operations and Observability

Metrics

  • Append latency and failure rate
  • Projection lag and throughput
  • Outbox pending count and publish retries

Tracing

  • Correlation ID from command to event to projection
  • Causation ID for event chains
  • Command ID for duplicate detection

Runbooks

  • Projection replay procedure
  • Dead-letter investigation process
  • Outbox backlog recovery steps
Section 12

Migration Strategy from CRUD

Step 1Capture domain events from writes (still CRUD source of truth).
Step 2Introduce outbox and projection handlers for new read models.
Step 3Move selected aggregates to event-sourced writes with version checks.
Migrate by bounded context, not by framework layer. Keep cut lines aligned with business capability boundaries.
Section 13

Common Failure Patterns

Anti-pattern Consequence Correction
Event schema as a copy of entity schema Breaks consumers whenever storage evolves Model events as business facts, not table snapshots
Shared database for all read models Tight coupling and replay risk Independent projection ownership and checkpoints
Putting domain rules in projection handlers Invariants enforced too late Keep invariants in aggregates only
Section 14

Implementation Roadmap

Phase Duration Deliverable
Foundation 2-3 weeks Event store schema, append/load API, aggregate test harness
Integration 2 weeks Outbox relay, idempotent publisher, correlation metadata
Read side 2-3 weeks Projection pipeline, checkpointing, replay tooling
Hardening 2 weeks Operational runbooks, lag alerts, DLQ recovery drills
Start with one bounded context end-to-end. Do not attempt full-domain migration in the first iteration.
