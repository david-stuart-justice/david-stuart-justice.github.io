Manual Event Sourcing with Spring Boot

Building event sourcing from scratch using Spring Boot, JPA, custom event tables, the outbox pattern, and message streaming. This document covers the approach, trade-offs, and how it compares to using a framework like Axon. It is exploratory research for learning purposes.

Context

What this document covers

Event sourcing without a framework means you own the event store schema, replay logic, outbox management, and read-side projection tracking. This approach gives maximum control and fine-grained optimization for your specific domain. It also means you build and maintain infrastructure that frameworks like Axon provide out of the box.

This document assumes a greenfield service with moderate transaction volume, Spring Boot backend, PostgreSQL storage, and message broker for downstream fan-out to independent consumers.

This is exploratory research for learning and comparison purposes.
High Level

Architecture overview

Commands update the event log and trigger the outbox. Outbox rows are published to the message broker asynchronously. Read-side consumers subscribe to the event stream and build projections independently. Unlike Axon, there is no automatic aggregate loading or optimistic concurrency checking — you implement these patterns for your domain.

Write side

  • Service receives command
  • Validates business rules
  • Writes event to event_store
  • Writes outbox row in same transaction
  • Returns response to caller

Asynchronous publisher

  • Polls outbox for unpublished rows
  • Publishes to message broker
  • Marks row as published
  • Retries on failure
  • Dead letters poison messages

Read side

  • Consumers receive stream records
  • Deserialize to domain events
  • Track sequence offsets
  • Build read-model projections
  • Handle duplicates via idempotency key
Schema

Event store design

The event store is a simple append-only table. Rows are never updated or deleted. Query patterns include "all events for aggregate X in order" and "all events since timestamp Y".

ColumnTypePurpose
id BIGSERIAL PRIMARY KEY Global insertion order. Monotonically increasing. Used for replaying events in sequence.
aggregate_id VARCHAR(255) NOT NULL The entity ID. Indexed for "all events for this aggregate" queries.
aggregate_type VARCHAR(100) NOT NULL Fully-qualified class name (e.g. com.example.OrderAggregate). Allows filtering by aggregate type.
event_type VARCHAR(100) NOT NULL Fully-qualified class name of the event (e.g. com.example.OrderPlacedEvent).
payload JSONB Serialized event data. Schema versioning via payload_version field inside the JSON.
sequence_number BIGINT NOT NULL Per-aggregate sequence, starting at 0. Unique constraint on (aggregate_id, sequence_number) for optimistic concurrency.
event_id UUID NOT NULL UNIQUE Global unique ID for the event. Used as idempotency key in downstream systems.
created_at TIMESTAMP DEFAULT NOW() Event creation time. Used for point-in-time queries.
metadata JSONB User, correlation ID, request origin, audit info — optional context around the event.

Create statement

CREATE TABLE event_store (
    id                BIGSERIAL PRIMARY KEY,
    aggregate_id      VARCHAR(255) NOT NULL,
    aggregate_type    VARCHAR(100) NOT NULL,
    event_type        VARCHAR(100) NOT NULL,
    payload           JSONB NOT NULL,
    sequence_number   BIGINT NOT NULL,
    event_id          UUID NOT NULL UNIQUE,
    created_at        TIMESTAMP DEFAULT NOW(),
    metadata          JSONB,
    UNIQUE(aggregate_id, sequence_number),
    INDEX idx_aggregate_id (aggregate_id),
    INDEX idx_created_at (created_at)
);
The unique constraint on (aggregate_id, sequence_number) provides optimistic concurrency. Two concurrent writes to the same aggregate will produce a constraint violation on the losing write. Your application detects this and can retry, reject, or escalate depending on the domain rules.
Reliability

Outbox pattern & Distributed Locking

The outbox table sits in the same database as the event store. Within a single transaction, you write the event and the outbox row. PostgreSQL's ACID guarantees ensure both succeed or both fail — no events are lost. An external relay process publishes outbox rows to the message broker asynchronously.

Clustered Deployment: When running multiple replicas of the publisher, you must ensure only one instance processes a specific set of outbox rows at a time to prevent duplicate broker records. Use a distributed lock (e.g., ShedLock) on the @Scheduled task.
1. Service writes event Event inserted into event_store.
2. Outbox row written In the same transaction, insert a row into outbox referencing the event.
3. Transaction commits Both rows are durable. If either fails, the entire transaction rolls back.
4. Relay publishes Separate async process polls outbox, publishes to message broker, marks as published.

Service method

@Service
public class OrderService {

    private final EntityManager em;
    private final BrokerProperties brokerConfig;

    @Transactional
    public void placeOrder(String customerId) {
        // Generate IDs
        String orderId = UUID.randomUUID().toString();
        String eventId = UUID.randomUUID().toString();

        // Create event payload
        OrderPlacedEvent event = new OrderPlacedEvent(
            orderId, customerId, Instant.now()
        );

        // Insert into event store
        em.createNativeQuery(
            "INSERT INTO event_store (aggregate_id, aggregate_type, event_type, " +
            "payload, sequence_number, event_id, metadata) " +
            "VALUES (?, 'com.example.OrderAggregate', ?, ?, 0, ?, ?)")
         .setParameter(1, orderId)
         .setParameter(2, event.getClass().getName())
         .setParameter(3, objectMapper.writeValueAsString(event))
         .setParameter(4, eventId)
         .setParameter(5, objectMapper.writeValueAsString(Map.of(
           "userId", getCurrentUserId(),
           "requestId", getCurrentRequestId()
         )))
         .executeUpdate();

        // Insert into outbox in the same transaction
        em.createNativeQuery(
            "INSERT INTO outbox (event_id, aggregate_id, event_type, payload) " +
            "VALUES (?, ?, ?, ?)"
        ).setParameter(1, eventId)
         .setParameter(2, orderId)
         .setParameter(3, event.getClass().getName())
         .setParameter(4, objectMapper.writeValueAsString(event))
         .executeUpdate();

        // Transaction commits atomically — both rows are durable
    }
}
Constraint on (aggregate_id, sequence_number). If two requests try to submit the same application (or two commands on the same aggregate execute concurrently), the second will fail on the unique constraint. The application must detect this and retry, reject, or merge the concurrent changes depending on domain rules.
Write Side

Command handling

Unlike Axon's aggregate pattern, you load events manually, apply business logic, and write the result. There is no automatic aggregate state reconstruction or optimistic concurrency handling — you implement both in the service layer.

Load aggregate from events

public OrderAggregate loadAggregate(String orderId) {
    // Fetch all events for this aggregate in order
    List<DomainEvent> events = em.createNativeQuery(
        "SELECT payload, event_type FROM event_store " +
        "WHERE aggregate_id = ? ORDER BY sequence_number ASC",
        Object[].class
    ).setParameter(1, applicationId)
     .getResultList()
     .stream()
     .map(row -> {
         String eventType = (String) row[1];
         String payload = (String) row[0];
         // Deserialize based on event type
         return objectMapper.readValue(payload, forName(eventType));
     })
     .collect(Collectors.toList());

    // Reconstruct aggregate state by replaying events
    OrderAggregate agg = new OrderAggregate(orderId);
    events.forEach(agg::apply);
    return agg;
}

// Aggregate class with apply() method
public class OrderAggregate {
    private String orderId;
    private OrderStatus status;

    public void apply(DomainEvent event) {
        if (event instanceof OrderPlacedEvent e) {
            this.status = OrderStatus.PLACED;
        } else if (event instanceof OrderApprovedEvent e) {
            this.status = OrderStatus.APPROVED;
        }
    }

    public void placeOrder(String customerId) {
        if (status != null) {
            throw new IllegalStateException("Order already exists");
        }
        apply(new OrderPlacedEvent(orderId, customerId, Instant.now()));
    }

    public void approve(String processedBy) {
        if (status != OrderStatus.PLACED) {
            throw new IllegalStateException("Cannot approve: status is " + status);
        }
        apply(new OrderApprovedEvent(orderId, processedBy, Instant.now()));
    }
}

Handle command with optimistic concurrency

@Transactional
public void approveOrder(String orderId, String processedBy) {
    // Load aggregate with current sequence number
    OrderAggregate agg = loadAggregateWithSequence(orderId);
    long nextSequence = agg.getSequenceNumber() + 1;

    // Apply command — this may throw if business rules are violated
    agg.approve(processedBy);
    OrderApprovedEvent event = agg.getUncommittedEvents().get(0);

    // Attempt to write the new event
    try {
        em.createNativeQuery(
            "INSERT INTO event_store " +
            "(aggregate_id, aggregate_type, event_type, payload, sequence_number, event_id) " +
            "VALUES (?, ?, ?, ?, ?, ?)"
        ).setParameter(1, orderId)
         .setParameter(2, "com.example.OrderAggregate"))
         .setParameter(3, event.getClass().getName())
         .setParameter(4, objectMapper.writeValueAsString(event))
         .setParameter(5, nextSequence)
         .setParameter(6, UUID.randomUUID().toString())
         .executeUpdate();
    } catch (DataIntegrityViolationException e) {
        // Unique constraint on (aggregate_id, sequence_number) was violated
        // Another command executed between our load and our write
        throw new OptimisticLockingException(orderId);
    }

    // Insert outbox row — same transaction, same guarantees
    em.createNativeQuery(
        "INSERT INTO outbox (event_id, aggregate_id, event_type, payload) " +
        "VALUES (?, ?, ?, ?)")
    .setParameter(1, event.getEventId())
     .setParameter(2, orderId)
     .setParameter(3, event.getClass().getName())
     .setParameter(4, objectMapper.writeValueAsString(event))
     .executeUpdate();
}
You own the load-modify-write cycle and concurrency handling. Unlike Axon, which wraps this pattern transparently, you see and manage the exception handling. This gives you more control but also more responsibility.
Reconstruction

Event replay & snapshots

Replay is straightforward: order events by creation time or global sequence, deserialize, and apply. For large event streams, implement snapshotting to avoid replaying hundreds of events on every load.

Full replay (for testing or debugging)

public void replayAllEvents(Consumer<DomainEvent> handler) {
    em.createNativeQuery(
        "SELECT payload, event_type FROM event_store ORDER BY id ASC",
        Object[].class
    ).getResultList().stream()
     .map(row -> {
         String eventType = (String) row[1];
         String payload = (String) row[0];
         return objectMapper.readValue(payload, forName(eventType));
     })
     .forEach(handler);
}

Snapshot table

Load with snapshot optimization

public OrderAggregate loadAggregateOptimized(String orderId) {
    // Check for snapshot
    var snapshot = em.createNativeQuery(
        "SELECT state, sequence_number FROM aggregate_snapshot " +
        "WHERE aggregate_id = ?",
        Object[].class
    ).setParameter(1, applicationId)
     .getResultList()
     .stream()
     .findFirst();

    OrderAggregate agg;
    long startSeq = 0;

    if (snapshot.isPresent()) {
        // Restore from snapshot
        Object[] row = (Object[]) snapshot.get();
        agg = objectMapper.readValue((String) row[0], ApplicationAggregate.class);
        startSeq = ((Number) row[1]).longValue() + 1;
    } else {
        agg = new ApplicationAggregate(applicationId);
    }

    // Replay only events after snapshot
    em.createNativeQuery(
        "SELECT payload, event_type FROM event_store " +
        "WHERE aggregate_id = ? AND sequence_number >= ? " +
        "ORDER BY sequence_number ASC",
        Object[].class
    ).setParameter(1, applicationId)
     .setParameter(2, startSeq)
     .getResultList().stream()
     .map(row -> objectMapper.readValue((String) row[0], forName((String) row[1])))
     .forEach(agg::apply);

    return agg;
}
Manual snapshots require explicit trigger logic — typically "create a snapshot every N events" or "on-demand when explicitly requested". You own the migration strategy if the snapshot format changes.
Read Side

Read-side projections

Consumers subscribe to the event stream and build their own projections. Each consumer maintains an offset or sequence number to track progress and handle restarts. A dedicated read model table holds the current state without needing to replay events on every query.

Consumer State

Tracking offsets & idempotency

Consumers track their progress via a sequence number or shard iterator. Storing this in the database allows resumption after restarts. Handling duplicates requires idempotency — applying the same event twice produces the same result.

Idempotent projection update

public void handleEvent(DomainEvent event) {
    // Upsert pattern: insert if new, update if already seen
    // The event_id acts as the idempotency key
    
    if (event instanceof ApplicationSubmittedEvent e) {
        jdbc.update(
            "INSERT INTO application_status_view " +
            "(application_id, status, applicant_id, updated_at) " +
            "VALUES (?, ?, ?, ?) " +
            "ON CONFLICT (application_id) DO UPDATE SET " +
            "status = ?, applicant_id = ?, updated_at = ?",
            e.getApplicationId(), "SUBMITTED", e.getApplicantId(), Instant.now(),
            "SUBMITTED", e.getApplicantId(), Instant.now()
        );
    }

    // Always update the offset after processing
    // If processing fails, offset is not updated, and the event is retried
    updateOffset(event.getEventId(), event.getSequenceNumber());
}

private void updateOffset(String eventId, long sequenceNumber) {
    jdbc.update(
        "INSERT INTO event_processing_log (event_id, processed_at) " +
        "VALUES (?, ?) ON CONFLICT (event_id) DO NOTHING",
        eventId, Instant.now()
    );
}
The idempotency log and the projection update are in the same database. A single transaction ensures both succeed or both fail. If the consumer crashes, it resumes from the last checkpointed offset and replays from there, re-idempotently processing events it has already seen.
Comparison

vs Axon: core differences

This manual approach and Axon solve the same problem — enabling event-driven architectures — but with different trade-offs. The table below highlights key architectural decisions.

ConcernManual ImplementationAxon Framework
Aggregate load from events Manual load + iterate + apply pattern Automatic via @EventSourcingHandler replay
Concurrency control Unique constraint on (aggregate_id, sequence_number); you handle exception Built-in, throws ConcurrencyException on collision
Outbox pattern Explicit table + scheduled relay process Structural: event store is the outbox; tracking processor is the relay
Read-side tracking Manual offset table per consumer Built-in token_entry per processor per segment
Event replay Manual SQL queries by aggregate or time Tracking token reset; framework handles position
Snapshotting Manual snapshot table; you trigger and restore Built-in @SnapshottingConfiguration with automatic triggers
Saga/long-running processes Custom state machine implementation @Saga with automatic persistence in saga_entry
Event schema evolution Manual migration or versioned handlers Built-in upcasting chain
Kinesis integration Direct AWS SDK call from outbox publisher No native extension; would use custom tracking processor
Testing aggregates Manual given-when-then test fixtures AggregateTestFixture without Spring context
Manual implementation gives you fine-grained control and no framework learning curve. Axon trades some flexibility for less boilerplate and automatic consistency checking. The right choice depends on domain complexity, team size, and long-term project scope.
Evaluation

Strategic Alignment

Manual implementation provides the highest degree of transparency and control, making it ideal for teams with strong SQL expertise who wish to avoid the "magic" of framework abstractions. However, it requires owning the infrastructure for concurrency, outbox relaying, and aggregate loading.
