Event Sourcing Research

Exploratory research into implementing event sourcing in a Spring Boot application. Two paths are examined: using Axon Framework for structure and consistency, or building from scratch with manual control and minimal dependencies. This is personal research, not a team decision.

Framework Approach

Axon Framework

A complete event sourcing and CQRS framework that provides aggregate loading, concurrency control, and tracking processor management.

  • Automatic state reconstruction from events
  • Built-in concurrency and conflict detection
  • Structural outbox via event store tracking
  • Advanced testing with AggregateTestFixture
Explore Axon approach
Manual Approach

Manual Event Sourcing

Building from scratch using Spring Boot, custom event tables, the transactional outbox pattern, and manual aggregate handling.

  • Fully transparent SQL-based implementation
  • Direct control over outbox and relay logic
  • Minimal "magic" and library dependencies
  • Optimistic locking via DB constraints

Project Constraints & Constraints

Scale & Technical Baseline

  • Volume: ~100,000 applications processed per year.
  • Infrastructure: AWS EKS (Kubernetes), Kinesis, PostgreSQL RDS.
  • Pipeline: GitHub Actions for CI/CD, Helm for deployments.
  • Stack: Java 25, Spring Boot 4, Flyway migrations.

Core Objectives

  • Auditability: Full history of state changes is mandatory.
  • Decoupling: Downstream services (e.g., Reporting) consumed via events.
  • Reliability: At-least-once delivery to Kinesis.
  • Maintainability: Code must be readable by generalist engineers.
