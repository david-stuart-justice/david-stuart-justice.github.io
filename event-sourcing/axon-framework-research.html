Context

What this document is

This is personal research into using Axon Framework as the basis for an event-sourced write model in a Spring Boot application backed by PostgreSQL. The starting point is an existing application with a JPA entity model and an event history table that records API requests as an audit log — events as a side effect of writes rather than as the source of truth.

The discussion explored whether Axon can replace that pattern with a structured event store, provide built-in projection and replay mechanisms, and connect to AWS Kinesis for downstream fan-out to independent read consumers.

This is not a team decision, ADR, or formal recommendation. It covers one explored path among several and would require validation with stakeholders before any commitment.
Overview

What Axon Framework provides

Axon is a Java framework for CQRS and event sourcing. It provides the command handling, event storing, read-side projection building, and process coordination that would otherwise require custom infrastructure. The core model is annotation-driven and integrates with Spring Boot via auto-configuration.

Write side

  • @Aggregate — domain object that enforces business rules
  • @CommandHandler — handles incoming commands and applies events
  • @EventSourcingHandler — rebuilds aggregate state by replaying events

Read side

  • @EventHandler — builds and updates read model projections
  • @QueryHandler — serves read queries from projections
  • Tracking processors with persistent position tokens in token_entry

Process coordination

  • @Saga — long-running processes that react to events and dispatch commands
  • Snapshot support for aggregates with long event histories
  • Built-in replay via token reset; no custom infrastructure required

The key conceptual shift from a JPA entity model: events become the source of truth. Current entity state is a derived projection, rebuilt from events on load. The audit table stops being a side effect and becomes the write model itself.

Configuration

Setup without Axon Server

axon-spring-boot-starter connects to Axon Server by default. Excluding the axon-server-connector module switches to a JPA-backed event store and in-process command and query buses. No additional infrastructure is required beyond a PostgreSQL database.

Write Side

The Aggregate

The aggregate is the consistency boundary on the write side. It holds business rules and emits events. It does not call any repository or external service. Loading an aggregate means replaying every stored event for that aggregate ID in sequence from domain_event_entry.

@Aggregate
public class ApplicationAggregate {

    @AggregateIdentifier
    private String applicationId;
    private ApplicationStatus status;
    private boolean caseworkerApproved;

    // Constructor-level @CommandHandler handles creation commands
    @CommandHandler
    public ApplicationAggregate(SubmitApplicationCommand cmd) {
        if (cmd.getApplicationId() == null || cmd.getApplicantId() == null) {
            throw new IllegalArgumentException("applicationId and applicantId are required");
        }
        AggregateLifecycle.apply(
            new ApplicationSubmittedEvent(cmd.getApplicationId(), cmd.getApplicantId())
        );
    }

    // @EventSourcingHandler rebuilds in-memory state — called on every load
    @EventSourcingHandler
    public void on(ApplicationSubmittedEvent event) {
        this.applicationId = event.getApplicationId();
        this.status = ApplicationStatus.SUBMITTED;
        this.caseworkerApproved = false;
    }

    @CommandHandler
    public void handle(ApproveApplicationCommand cmd) {
        if (status != ApplicationStatus.SUBMITTED) {
            throw new IllegalStateException(
                "Cannot approve: application is in state " + status
            );
        }
        AggregateLifecycle.apply(
            new ApplicationApprovedEvent(applicationId, cmd.getApproverId())
        );
    }

    @EventSourcingHandler
    public void on(ApplicationApprovedEvent event) {
        this.status = ApplicationStatus.APPROVED;
        this.caseworkerApproved = true;
    }

    @CommandHandler
    public void handle(WithdrawApplicationCommand cmd) {
        if (status == ApplicationStatus.WITHDRAWN || status == ApplicationStatus.COMPLETE) {
            throw new IllegalStateException(
                "Cannot withdraw: application is in state " + status
            );
        }
        AggregateLifecycle.apply(
            new ApplicationWithdrawnEvent(applicationId, cmd.getReason())
        );
    }

    @EventSourcingHandler
    public void on(ApplicationWithdrawnEvent event) {
        this.status = ApplicationStatus.WITHDRAWN;
    }
}
The aggregate has no @Autowired dependencies. It holds only the state needed to enforce business rules — not state for display or querying. AggregateLifecycle.apply() both stores the event and immediately calls the corresponding @EventSourcingHandler to update in-memory state.
Infrastructure

Event store schema

Axon creates these tables automatically on startup via JPA. They are the backbone of both write-side consistency and read-side replay.

TablePurposeKey columns
domain_event_entry All events for all aggregates. The primary source of truth. global_index, aggregate_identifier, sequence_number, type, payload_type, payload (JSON), event_identifier, time_stamp, meta_data
snapshot_event_entry Aggregate snapshots, written periodically to avoid replaying long event streams on every load. aggregate_identifier, sequence_number, type, payload
token_entry Tracking processor positions. One row per processor per segment. Equivalent to a Kinesis sequence number checkpoint. processor_name, segment, token (serialized position), owner
saga_entry Serialized state for in-flight Sagas. Persists across service restarts. saga_id, saga_type, serialized_saga, revision
association_value_entry Saga association properties, used to route events to the correct Saga instance by a property such as applicationId. saga_id, association_key, association_value
dead_letter_entry Events that failed processing and were quarantined. Created if a dead letter queue is configured. processing_group, sequence_identifier, message_identifier, cause_type, cause_message
Optimistic concurrency: the pair (aggregate_identifier, sequence_number) has a unique constraint in domain_event_entry. Two concurrent commands on the same aggregate will produce a constraint violation on the losing write. Axon surfaces this as a ConcurrencyException and can be configured to retry.

Schema Migration (Flyway SQL)

Using spring.jpa.hibernate.ddl-auto=validate requires providing the schema via Flyway. The core Axon tables for a JPA event store are domain_event_entry and snapshot_event_entry.

-- V1__axon_initial_schema.sql
CREATE TABLE domain_event_entry (
    global_index      BIGSERIAL PRIMARY KEY,
    event_identifier  VARCHAR(255) NOT NULL UNIQUE,
    meta_data         BYTEA,
    payload           BYTEA NOT NULL,
    payload_revision  VARCHAR(255),
    payload_type      VARCHAR(255) NOT NULL,
    time_stamp        VARCHAR(255) NOT NULL,
    aggregate_identifier VARCHAR(255) NOT NULL,
    sequence_number   BIGINT NOT NULL,
    type              VARCHAR(255),
    UNIQUE (aggregate_identifier, sequence_number)
);

CREATE TABLE snapshot_event_entry (
    aggregate_identifier VARCHAR(255) NOT NULL,
    sequence_number      BIGINT NOT NULL,
    type                 VARCHAR(255) NOT NULL,
    event_identifier     VARCHAR(255) NOT NULL UNIQUE,
    meta_data            BYTEA,
    payload              BYTEA NOT NULL,
    payload_revision     VARCHAR(255),
    payload_type         VARCHAR(255) NOT NULL,
    time_stamp           VARCHAR(255) NOT NULL,
    PRIMARY KEY (aggregate_identifier, sequence_number, type)
);

CREATE TABLE token_entry (
    processor_name VARCHAR(255) NOT NULL,
    segment        INTEGER NOT NULL,
    owner          VARCHAR(255),
    timestamp      VARCHAR(255) NOT NULL,
    token          BYTEA,
    token_type     VARCHAR(255),
    PRIMARY KEY (processor_name, segment)
);
API Layer

Dispatching commands from the API

Spring MVC controllers dispatch commands through CommandGateway rather than calling a service method. The controller's responsibility is to translate the HTTP request into a command object and return an appropriate response. Routing to the correct aggregate instance is handled by the framework.

@RestController
@RequestMapping("/applications")
public class ApplicationController {

    private final CommandGateway commandGateway;

    public ApplicationController(CommandGateway commandGateway) {
        this.commandGateway = commandGateway;
    }

    @PostMapping
    public ResponseEntity<String> submit(@RequestBody SubmitApplicationRequest request) {
        String applicationId = UUID.randomUUID().toString();
        commandGateway.sendAndWait(
            new SubmitApplicationCommand(applicationId, request.getApplicantId())
        );
        return ResponseEntity.status(201).body(applicationId);
    }

    @PostMapping("/{id}/approve")
    public ResponseEntity<Void> approve(
            @PathVariable String id,
            @RequestBody ApproveRequest request) {
        commandGateway.sendAndWait(
            new ApproveApplicationCommand(id, request.getApproverId())
        );
        return ResponseEntity.noContent().build();
    }

    @PostMapping("/{id}/withdraw")
    public ResponseEntity<Void> withdraw(
            @PathVariable String id,
            @RequestBody WithdrawRequest request) {
        commandGateway.sendAndWait(
            new WithdrawApplicationCommand(id, request.getReason())
        );
        return ResponseEntity.noContent().build();
    }
}

sendAndWait blocks until the command handler completes and the event is committed to the event store. send returns a CompletableFuture. sendAndWait is appropriate for request-response APIs where the caller needs confirmation before returning. Business rule violations thrown by the aggregate propagate back as exceptions through CommandGateway.

Read Side

Projections with tracking processors

Tracking processors poll domain_event_entry using a position stored in token_entry. On restart they resume from the last committed position. Resetting the token triggers a replay from that point. Each @ProcessingGroup has its own independent token.

Two modes exist for event processors. Tracking processors (recommended) poll the event store at their own pace, maintain a durable position token, and can replay from any point. Subscribing processors receive events synchronously in the same thread as the command handler — simpler, but not replayable and tightly coupled to the write path.

@Component
@ProcessingGroup("application-status")
public class ApplicationStatusProjection {

    private final ApplicationStatusRepository repo;

    public ApplicationStatusProjection(ApplicationStatusRepository repo) {
        this.repo = repo;
    }

    @EventHandler
    public void on(ApplicationSubmittedEvent event) {
        repo.save(new ApplicationStatusView(
            event.getApplicationId(),
            "SUBMITTED",
            event.getApplicantId(),
            Instant.now()
        ));
    }

    @EventHandler
    public void on(ApplicationApprovedEvent event) {
        repo.findById(event.getApplicationId()).ifPresent(view -> {
            view.setStatus("APPROVED");
            view.setLastUpdated(Instant.now());
            repo.save(view);
        });
    }

    @EventHandler
    public void on(ApplicationWithdrawnEvent event) {
        repo.findById(event.getApplicationId()).ifPresent(view -> {
            view.setStatus("WITHDRAWN");
            view.setWithdrawalReason(event.getReason());
            view.setLastUpdated(Instant.now());
            repo.save(view);
        });
    }
}
Idempotency is required. Tracking processors provide at-least-once delivery. If a handler succeeds but the service crashes before the token commits, the event is redelivered on restart. Every @EventHandler method must produce the same result when called twice with the same event — a safe pattern is to use upsert operations (save-or-update) rather than unconditional inserts.

Query handling

Axon provides a QueryGateway and @QueryHandler to serve reads through the same framework pattern as commands. This is optional — controllers can call the read model repository directly to start with, and the query bus can be introduced later when it adds value (for example, when multiple services need to query the same model).

// Declare a query object
public record GetApplicationStatusQuery(String applicationId) {}

// Add a @QueryHandler to the projection class
@QueryHandler
public ApplicationStatusView handle(GetApplicationStatusQuery query) {
    return repo.findById(query.getApplicationId())
        .orElseThrow(() -> new ApplicationNotFoundException(query.getApplicationId()));
}

// Dispatch from the controller
@GetMapping("/{id}/status")
public ApplicationStatusView getStatus(@PathVariable String id) {
    return queryGateway.query(
        new GetApplicationStatusQuery(id),
        ResponseTypes.instanceOf(ApplicationStatusView.class)
    ).join();
}
Event Replay

Replay options

Axon's replay mechanism works by resetting the tracking token for a processor. The processor re-reads events from the event store from that position. This is the mechanism for rebuilding a corrupted read model, migrating to a new projection schema, or debugging what happened at a point in time.

Use caseMechanismScope
Rebuild entire read model from scratch Reset processor token to tail (beginning of event store) All events, all aggregates
Rebuild read model from a specific date Reset token to a TrackingToken at a given Instant All events from that timestamp forward
Process only future events (skip history) Reset token to head (current end of event store) Events written after the reset only
Read all events for one aggregate eventStore.readEvents(aggregateId) All events for a single aggregate ID, in sequence
Reconstruct state of one aggregate at date X Read events by ID, filter by timestamp, apply manually Single aggregate up to target point in time 
// Obtain a reference to the tracking processor by its processing group name
TrackingEventProcessor processor = eventProcessingConfiguration
    .eventProcessorByProcessingGroup("application-status", TrackingEventProcessor.class)
    .orElseThrow(() -> new IllegalStateException("Processor not found"));

processor.shutDown();

// Option 1: full replay from the beginning of the event store
processor.resetTokens();

// Option 2: replay from a specific point in time (all aggregates from that date forward)
TrackingToken tokenAt = eventStore.createTokenAt(
    Instant.parse("2026-01-01T00:00:00Z")
);
processor.resetTokens(tokenAt);

// Option 3: skip all history, process only future events
processor.resetTokens(eventStore.createHeadToken());

processor.start();

Point-in-time state for a single aggregate

Tracking processor replay is always global — it processes all events across all aggregates in order. There is no built-in mechanism to replay only the events for one aggregate ID. For reconstructing the state of a specific application at a given date, use the event store API directly.

// Reconstruct the state of one application as it existed on a given date
Instant targetDate = Instant.parse("2026-03-15T12:00:00Z");

DomainEventStream events = eventStore.readEvents(applicationId);
List<DomainEventMessage<?>> relevantEvents = events.asStream()
    .filter(event -> !event.getTimestamp().isAfter(targetDate))
    .collect(Collectors.toList());

Snapshotting

Loading an aggregate with hundreds of events requires replaying all of them. Snapshotting writes the current aggregate state into snapshot_event_entry after a threshold number of events. Subsequent loads start from the snapshot and replay only the events that followed it.

// build.gradle — no additional dependency needed, snapshotter is included

// Java configuration
@Bean
public SnapshottingConfiguration snapshottingConfiguration(
        SnapshottingConfigurer configurer,
        Snapshotter snapshotter) {
    return configurer
        .forType(ApplicationAggregate.class)
        .snapshotTriggerDefinition(
            new EventCountSnapshotTriggerDefinition(snapshotter, 50)
        )
        .andConfigure();
}
If snapshotting is active and you need to reconstruct state at a point in time before the latest snapshot, you must bypass the snapshot and read raw events from domain_event_entry directly. The snapshot represents a point after which earlier event history is no longer traversed on normal load.
Reliability

Built-in outbox behaviour

Axon does not call it an outbox, but the event store combined with tracking processors provides the same guarantees without any additional infrastructure. The event is written to PostgreSQL within the aggregate's local transaction. The downstream action — updating a read model, publishing to Kinesis — happens separately via a tracking processor that only advances its token after the action succeeds.

1. Command handled Aggregate applies event. Axon writes it to domain_event_entry atomically within the transaction.
2. Processor reads event Tracking processor polls domain_event_entry and reads the committed event.
3. Handler executes @EventHandler runs — updates the read model, publishes to Kinesis, or any other action.
4. Token advances Only on successful handler execution does Axon commit the updated token to token_entry.

If the downstream system is unavailable, the token does not advance. The processor stalls or retries. When the downstream system recovers, the processor resumes from the last committed token position. No events are lost — they are durable in PostgreSQL before the processor ever runs.

A traditional outbox pattern requires building an outbox table, a relay process, and failure handling explicitly. Axon's design makes this structural. The event store is the outbox; the tracking processor is the relay.

At-least-once delivery: if the handler succeeds but the service restarts before the token commits, the event is redelivered. Handlers must be idempotent. Axon generates a unique event_identifier (UUID) for every event — use this as the idempotency key in downstream systems.

Dead letter queue for poison events

By default, a consistently failing event blocks all subsequent events for that processor. Configuring a dead letter queue quarantines failing events and allows the processor to continue with the rest of the stream.

// build.gradle — no additional dependency needed

// Java configuration
@Autowired
public void configure(EventProcessingConfigurer configurer) {
    configurer.registerDeadLetterQueue(
        "application-status",    // processing group name
        config -> JpaSequencedDeadLetterQueue.builder()
            .processingGroup("application-status")
            .maxSequences(256)
            .maxSequenceSize(256)
            .transactionManager(config.getComponent(TransactionManager.class))
            .entityManagerProvider(config.getComponent(EntityManagerProvider.class))
            .serializer(config.serializer())
            .build()
    );
}
Audit

Full payload audit trail

Axon events should carry only the data needed to reconstruct aggregate state and update projections. Embedding full API request payloads in events makes them expensive to replay and difficult to evolve. The full request payload is a compliance concern, not an event sourcing concern. The two should be handled independently and linked by reference.

Option 1 — Command interceptor + S3

A MessageDispatchInterceptor writes the full request payload to S3 before the command is dispatched. It attaches the resulting S3 key to the command's metadata. That key propagates into the resulting event's metadata automatically, linking every domain event back to its raw request.

@Bean
public MessageDispatchInterceptor<CommandMessage<?>>
        auditInterceptor(AuditStore auditStore) {
    return messages -> (index, msg) -> {
        String s3Key = auditStore.store(
            msg.getPayloadType().getSimpleName(),
            msg.getPayload()
        );
        return msg.andMetaData(
            Map.of("auditKey", s3Key,
                   "auditTimestamp", Instant.now().toString())
        );
    };
}

// Register the interceptor with the command bus
@Autowired
public void registerInterceptor(
        CommandBus commandBus,
        MessageDispatchInterceptor<CommandMessage<?>> auditInterceptor) {
    commandBus.registerDispatchInterceptor(auditInterceptor);
}

Option 2 — Separate audit log

A servlet filter writes the full request to an append-only audit table or S3 before the command is dispatched. Axon has no knowledge of it. The two stores are linked only by a correlation ID passed as an HTTP header on every request.

  • Simpler — no Axon-specific code required
  • Covers read requests as well as writes
  • Correlation ID links audit record to domain event via metadata
  • Audit log can be written to S3 directly from the filter using AWS SDK v2
This option keeps the two concerns completely independent. The audit log and the event store are separate systems. Either can evolve without affecting the other.

In both options, Axon events remain small and domain-focused. The S3 audit log holds the raw compliance record. They are linked by a key or correlation ID, not by embedding payloads in events.

Consistency

Business rules and state guards

All rules governing whether a command is valid sit inside the aggregate's @CommandHandler methods. The aggregate holds whatever state is needed to enforce those rules. That state is not available for external querying — it exists solely so the aggregate can accept or reject commands.

@CommandHandler
public void handle(CompleteApplicationCommand cmd) {
    // State guard: caseworker must have approved before an application can complete
    if (!caseworkerApproved) {
        throw new CaseworkerApprovalRequiredException(applicationId);
    }
    if (status == ApplicationStatus.COMPLETE) {
        throw new IllegalStateException("Application is already complete");
    }
    AggregateLifecycle.apply(
        new ApplicationCompletedEvent(applicationId, cmd.getCompletedAt())
    );
}

@EventSourcingHandler
public void on(ApplicationCompletedEvent event) {
    this.status = ApplicationStatus.COMPLETE;
}
Commands that violate business rules throw before apply() is called. No event is written. The aggregate state does not change. The caller receives the exception through CommandGateway. The aggregate's internal state fields (caseworkerApproved, status) are not queryable from outside — the read model is the source of truth for reads.
Long-running Processes

Sagas

A Saga coordinates a multi-step business process that spans multiple events and commands over time. It reacts to events, maintains its own state, and dispatches commands when conditions are met. Saga state is serialized into saga_entry after each event, so it survives service restarts.

A representative use case: completing an application requires caseworker approval and document verification. These may occur in any order and at different times. The Saga listens for both events and dispatches CompleteApplicationCommand only when both have been received.

@Saga
public class ApplicationCompletionSaga {

    @Inject
    private transient CommandGateway commandGateway;

    private boolean caseworkerApproved = false;
    private boolean documentsVerified  = false;

    // Saga is created when an application is submitted
    @StartSaga
    @SagaEventHandler(associationProperty = "applicationId")
    public void on(ApplicationSubmittedEvent event) {
        // Saga initialised — awaiting conditions
    }

    @SagaEventHandler(associationProperty = "applicationId")
    public void on(CaseworkerApprovedEvent event) {
        caseworkerApproved = true;
        completeIfReady(event.getApplicationId());
    }

    @SagaEventHandler(associationProperty = "applicationId")
    public void on(DocumentsVerifiedEvent event) {
        documentsVerified = true;
        completeIfReady(event.getApplicationId());
    }

    // Saga ends when the application reaches the complete state
    @EndSaga
    @SagaEventHandler(associationProperty = "applicationId")
    public void on(ApplicationCompletedEvent event) {
        // Saga state removed from saga_entry
    }

    // Handle withdrawal — no point waiting for completion conditions
    @EndSaga
    @SagaEventHandler(associationProperty = "applicationId")
    public void on(ApplicationWithdrawnEvent event) {
        // Clean up without dispatching
    }

    private void completeIfReady(String applicationId) {
        if (caseworkerApproved && documentsVerified) {
            commandGateway.send(
                new CompleteApplicationCommand(applicationId, Instant.now())
            );
        }
    }
}
The Saga coordinates when to issue CompleteApplicationCommand. The aggregate still enforces whether the command is valid — the two concerns remain separate. The Saga dispatches the command; the aggregate's @CommandHandler rejects it if business rules are not met.
Read Models

Independent read model statuses

Multiple tracking processors can consume the same event stream independently and maintain different status representations. Each processor has its own token in token_entry and runs at its own pace. A failure or replay in one does not affect the others.

A canonical status projection holds the authoritative lifecycle state enforced by the aggregate's business rules. Domain-specific projections derive their own status representations from the same events, shaped for a specific consumer's workflow.

Canonical status projection

One authoritative record of the application's current lifecycle state. Derived from events produced by the aggregate, which enforces the transitions.

@Component
@ProcessingGroup("canonical-status")
public class CanonicalStatusProjection {

    private final CanonicalStatusRepository repo;

    @EventHandler
    public void on(ApplicationSubmittedEvent e) {
        repo.save(new CanonicalStatus(
            e.getApplicationId(), "SUBMITTED"
        ));
    }

    @EventHandler
    public void on(ApplicationApprovedEvent e) {
        repo.findById(e.getApplicationId())
            .ifPresent(s -> {
                s.setStatus("APPROVED");
                repo.save(s);
            });
    }

    @EventHandler
    public void on(ApplicationCompletedEvent e) {
        repo.findById(e.getApplicationId())
            .ifPresent(s -> {
                s.setStatus("COMPLETE");
                repo.save(s);
            });
    }
}

Caseworker dashboard projection

A separate projection that derives its own status vocabulary from the same events — reflecting workflow steps meaningful to caseworkers that do not belong in the canonical lifecycle.

@Component
@ProcessingGroup("caseworker-dashboard")
public class CaseworkerDashboardProjection {

    private final DashboardViewRepository repo;

    @EventHandler
    public void on(ApplicationSubmittedEvent e) {
        repo.save(new DashboardView(
            e.getApplicationId(), "AWAITING_ASSIGNMENT"
        ));
    }

    @EventHandler
    public void on(ApplicationAssignedEvent e) {
        repo.findById(e.getApplicationId())
            .ifPresent(v -> {
                v.setStatus("IN_REVIEW");
                v.setAssignee(e.getAssigneeId());
                repo.save(v);
            });
    }

    @EventHandler
    public void on(DocumentsVerifiedEvent e) {
        repo.findById(e.getApplicationId())
            .ifPresent(v -> {
                v.setStatus("READY_TO_COMPLETE");
                repo.save(v);
            });
    }
}
Both processors read from the same domain_event_entry table. Neither is aware of the other. Resetting the token for caseworker-dashboard triggers a replay for that projection only — the canonical status projection is unaffected.
External Distribution

Kinesis integration

There is no native Axon Kinesis extension. The standard approach is a dedicated tracking processor that reads committed events from the event store and publishes them to Kinesis. Events are durably stored in PostgreSQL before the processor runs — the same structural guarantee as the outbox pattern.

// build.gradle — add AWS SDK for Kinesis
dependencies {
    implementation 'software.amazon.awssdk:kinesis:2.25.0'
    // ... other dependencies
}

// Kinesis publisher tracking processor
@Component
@ProcessingGroup("kinesis-publisher")
public class KinesisEventPublisher {

    private final KinesisAsyncClient kinesis;
    private final ObjectMapper objectMapper;
    private final String streamName;

    public KinesisEventPublisher(KinesisAsyncClient kinesis,
                                  ObjectMapper objectMapper,
                                  @Value("${kinesis.stream.name}") String streamName) {
        this.kinesis = kinesis;
        this.objectMapper = objectMapper;
        this.streamName = streamName;
    }

    @EventHandler
    public void on(ApplicationSubmittedEvent event,
                   @MessageIdentifier String eventId) {
        publish(eventId, event.getApplicationId(), event);
    }

    @EventHandler
    public void on(ApplicationApprovedEvent event,
                   @MessageIdentifier String eventId) {
        publish(eventId, event.getApplicationId(), event);
    }

    @EventHandler
    public void on(ApplicationCompletedEvent event,
                   @MessageIdentifier String eventId) {
        publish(eventId, event.getApplicationId(), event);
    }

    private void publish(String eventId, String partitionKey, Object payload) {
        try {
            byte[] data = objectMapper.writeValueAsBytes(Map.of(
                "eventId", eventId,
                "eventType", payload.getClass().getSimpleName(),
                "payload", payload
            ));
            PutRecordRequest request = PutRecordRequest.builder()
                .streamName(streamName)
                .partitionKey(partitionKey)   // applicationId ensures ordering per application
                .data(SdkBytes.fromByteArray(data))
                .build();
            kinesis.putRecord(request).join();
        } catch (JsonProcessingException e) {
            throw new RuntimeException("Failed to serialize event " + eventId, e);
        }
    }
}
At-least-once delivery to Kinesis. Kinesis consumers must handle duplicates using event_identifier as the idempotency key. Configure a dead letter queue for the kinesis-publisher processing group so a consistently failing event does not stall publication for all subsequent events. Partitioning by applicationId ensures ordering per application within a Kinesis shard.
Alternative Approach

Kafka instead of Kinesis

Axon has a native Kafka extension that removes the need for a custom tracking processor publisher. On EKS, Kafka can run via the Strimzi operator or Amazon MSK. Click to expand pros, cons, and deployment notes.

The axon-kafka-spring-boot-starter extension bridges the Axon event store and a Kafka topic. Events committed to domain_event_entry are automatically forwarded to Kafka by the extension. Downstream consumers — Axon-based or otherwise — subscribe to the topic independently, with no knowledge of Axon required.

build.gradle

dependencies {
    implementation('org.axonframework:axon-spring-boot-starter:4.10.3') {
        exclude group: 'org.axonframework', module: 'axon-server-connector'
    }
    // Version must match the axon-spring-boot-starter version
    implementation 'org.axonframework.extensions.kafka:axon-kafka-spring-boot-starter:4.0.0'
    implementation 'org.springframework.boot:spring-boot-starter-data-jpa'
    implementation 'org.springframework.boot:spring-boot-starter-web'
    runtimeOnly 'org.postgresql:postgresql'
}

application.properties

axon.axonserver.enabled=false
axon.kafka.bootstrap-servers=my-kafka-cluster:9092
axon.kafka.publisher.confirmation-mode=transactional
axon.kafka.producer.transaction-id-prefix=axon-producer-
axon.kafka.default-topic=application-events

Deployment on EKS

Two options when running on EKS with Helm:

Strimzi operator (self-hosted)

Kafka runs inside the cluster as Kubernetes resources managed by the Strimzi operator. A Helm chart is available and deploys alongside other application Helm charts.

helm repo add strimzi https://strimzi.io/charts/
helm install strimzi-kafka-operator \
  strimzi/strimzi-kafka-operator \
  --namespace kafka --create-namespace
  • Kafka stays within the cluster — no external managed service dependency
  • Full control over retention, topic configuration, and upgrades
  • Adds operational responsibility for the broker itself

Amazon MSK (managed)

AWS-managed Kafka. No broker pods in EKS. Applications running on EKS connect to MSK over the VPC using IAM authentication via IRSA.

  • No broker operational overhead — AWS manages availability and patching
  • IAM-based auth integrates naturally with EKS IRSA
  • More expensive than Kinesis for low-throughput workloads
  • Some Kafka features are restricted compared to self-hosted

GitHub Actions — local testing with Testcontainers

Testcontainers provides an embedded Kafka container that runs inside GitHub Actions workflows without any external broker. This is a significant advantage over Kinesis, which requires either LocalStack or mocking for local and CI testing.

Pros and cons vs Kinesis

Advantages over Kinesis

  • Native Axon extension — no custom tracking processor publisher to write or maintain
  • Configurable retention (hours to indefinite, vs Kinesis 1–365 days)
  • Consumer group offset reset provides native replay from any position without custom logic
  • Testcontainers Kafka image works in GitHub Actions without LocalStack or mocking
  • Richer tooling: Kafka UI, Schema Registry (Confluent or AWS Glue), kcat for inspection
  • Strimzi keeps everything inside EKS — no dependency on a second AWS streaming service

Disadvantages vs Kinesis

  • At ~2–5 events per second peak, Kinesis is sufficient and operationally lighter than self-hosted Kafka
  • MSK costs more than Kinesis for this throughput level; the volume does not justify the price difference
  • Kinesis integrates natively with Lambda, Firehose, S3, and CloudWatch without additional connectors
  • The Axon Kafka extension adds its own configuration surface (producer transactions, topic naming, consumer group setup)
  • At-least-once delivery and idempotency requirements are identical — Kafka does not remove this concern
At ~100,000 applications per year, Kinesis is the simpler AWS-native choice. Kafka becomes more compelling if throughput grows significantly, if longer native retention without an S3 archive is needed, or if non-AWS consumers need to subscribe without going through a Kinesis-specific SDK. The native Axon Kafka extension is a meaningful advantage if the team is already running Kafka elsewhere in the platform.
Assessment

Trade-offs summary

Axon takes on infrastructure that would otherwise require custom implementation. The cost is an opinionated programming model and a meaningful learning curve. The table below maps key concerns to what each approach requires.

ConcernCustom implementationWith Axon
Event storage Custom event history table, manually designed schema Auto-created domain_event_entry via JPA
Aggregate load from event history Custom replay logic per aggregate Built-in via @EventSourcingHandler replay
Optimistic concurrency control Custom version column or JPA @Version Sequence number unique constraint, ConcurrencyException on collision
Read-side tracking position Custom checkpoint table (equivalent to Kinesis sequence number) token_entry, managed automatically by tracking processor
Projection replay Custom re-read and reprocess logic per projection Token reset via framework API; processor handles replay automatically
Multi-step business processes Custom process table or state machine implementation @Saga, persisted in saga_entry, survives restarts
Outbox pattern Explicit outbox table + relay process + failure handling Structural: event store is the outbox; tracking processor is the relay
Event schema evolution Manual migration scripts or versioned handlers Built-in upcasting chain — old event formats transformed before handler receives them
Distributed command routing (multi-service) Manual routing via messaging infrastructure In-process only without Axon Server; Spring Cloud extension needed for distribution
Kinesis integration Direct producer call from service layer Dedicated tracking processor publisher; no native extension exists
Unit testing command handling Manual test fixtures, Spring context or mocks required AggregateTestFixture — given/when/then without a Spring context
The main constraint without Axon Server is that command routing is in-process only. Aggregates in the same Spring Boot application can handle any command; aggregates in separate services require either Axon Server or the Spring Cloud extension. For a single-service write model with Kinesis-based fan-out to independent consumers, Axon without Axon Server is a reasonable fit.
